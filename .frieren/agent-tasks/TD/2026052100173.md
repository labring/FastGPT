#### 【问题】
训练模型（生成新模型）弹窗中，含有"处理中"文件的知识库（`dataset_trainings` 集合中存在记录）可以被选中并提交训练任务。

#### 【问题原因】
`BaseModelTrainModal.tsx` 渲染知识库列表时，直接展示所有知识库，未检查其是否含有正在处理中（训练队列存在）的集合，导致用户可以选中并发起训练。

根因：`DatasetListItemType` 没有 `processingCount` 字段，前端无法感知知识库的处理状态。

**追加问题**：`listWithChildren.ts` 中 aggregate 查询 `$in` 参数类型不匹配——post-find hook 将 `_id` 转为字符串后直接传入 aggregate，导致 `dataset_trainings` 查询始终返回空。

#### 【修复方案】
1. **`packages/global/core/dataset/type.ts`**：在 `DatasetListItemSchema` 中新增 `processingCount?: number` 字段，表示该知识库在训练队列中待处理的文件数量。

2. **`projects/app/src/pages/api/core/dataset/listWithChildren.ts`** 和 **`list.ts`**：通过聚合查询 `dataset_trainings` 集合，统计每个非文件夹知识库的训练队列记录数，填充到 `processingCount` 字段返回。

3. **【追加修复】`listWithChildren.ts` aggregate `$in` 类型不匹配问题**：
   - **根因**：项目全局 `addCommonMiddleware` 中 `post(/^find/)` 钩子会将 `.lean()` 返回的文档中的 `_id` 从 ObjectId 转为 string。`listWithChildren.ts` 直接将转后的字符串 ID 传给 aggregate 的 `$in`，导致 MongoDB aggregate pipeline 无法匹配 ObjectId 类型的字段（`dataset_trainings.datasetId` 是 ObjectId），返回空结果，`processingCount` 始终为 0。
   - **修复**：在 `listWithChildren.ts` 中，将 `datasetObjectIds = uniqueNonFolderDatasets.map((d) => d._id)` 改为 `datasetObjectIds = uniqueNonFolderDatasets.map((d) => new Types.ObjectId(d._id))`，与 `list.ts`（第 179 行）保持一致。

4. **`projects/app/src/components/core/ai/ModelTable/BaseModelTrainModal.tsx`**：
   - 引入 `MyTooltip`
   - 对 `processingCount > 0` 的知识库，禁用 Checkbox（`isDisabled`）、禁用点击行为（`cursor: not-allowed`）、降低不透明度（`opacity: 0.5`）
   - 悬浮时显示 Tooltip：`"仍在处理中，暂不支持训练"`
   - 全选/文件夹全选时过滤掉处理中的知识库
   - 文件夹半选/全选状态计算时排除处理中的知识库（分母只计算可选项）

4. **i18n**（三语种）：新增词条 `account_model:dataset_still_processing_tip`。

#### 【影响模块】
- `BaseModelTrainModal.tsx`：训练模型（生成新模型）弹窗知识库选择
- `DatasetListItemType`：全量知识库列表类型，所有使用该类型的场景均可感知 `processingCount`

#### 【测试建议】
1. 知识库中存在"处理中"文件时，打开训练弹窗，该知识库应显示为禁用状态，无法勾选，悬浮显示"仍在处理中，暂不支持训练"
2. 知识库中所有文件处理完成后，该知识库应恢复为可选状态
3. 点击"全选"时，处理中的知识库不应被选中
4. 勾选含处理中子项的文件夹时，处理中的知识库不应被选中；文件夹的半选/全选状态应正确反映可选子项的选中状态
5. 正常知识库（无处理中文件）选择、提交训练流程无回归
