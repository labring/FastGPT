## 节点概述

【**批量执行（并行）**】节点用于对数组输入进行并发处理：系统会为数组中的每个元素创建独立任务并行执行，最后按输入下标顺序聚合结果。

它适合“每个元素互不依赖”的批处理任务，相比串行批量执行通常有更高吞吐。

> 【建议配图 1】放在本节末尾：并行批量执行节点卡片 + 基础参数面板（数组、并发上限、单轮报错重试次数），用于让用户先建立整体认知。

## 核心特性

1. **并行执行能力**
   - 输入数组后并发处理多个元素
   - 实际并发度可配置（并发上限）
   - 每个元素可独立重试（单轮报错重试次数）

2. **结果可追踪**
   - 输出成功结果数组（仅包含成功项）
   - 输出原始结果数组（成功/失败逐项明细）
   - 输出整体状态（`success`/`failed`/`partial_success`）

3. **任务隔离**
   - 每个元素在独立运行时上下文中执行
   - 单个元素失败不会中断其他元素
   - 最终结果按输入数组顺序稳定输出

## 与串行批量执行共通的优势

虽然执行方式不同，但以下优势在并行模式下依然成立：

1. **统一的循环体编排体验**
   - 都通过“开始/结束”定义子流程
   - 都支持在循环体内复用已有节点能力（AI、HTTP、文本处理等）
   - 都能在结束节点明确指定每轮输出变量

2. **统一的结果聚合能力**
   - 都支持将每轮输出收集为数组，便于下游节点继续处理
   - 都可通过结构化输出提升可调试性与可维护性

3. **统一的容错思路**
   - 都强调“单项失败可定位、整体结果可判断”
   - 都支持通过原始结果明细快速排查失败元素

## 适用场景

【**批量执行（并行）**】节点特别适合以下场景：

1. **高吞吐批处理**
   - 批量翻译短文本
   - 批量摘要
   - 批量内容结构化提取

2. **独立任务并发**
   - 多条检索结果的并行分析
   - 多个 API 请求结果的并行加工
   - 多段文本的并行改写

3. **容错优先流程**
   - 允许部分元素失败
   - 继续处理剩余元素并汇总可用结果
   - 通过原始结果定位失败原因

## 并行模式适用边界

1. **强烈推荐使用并行**
   - 单个元素之间没有前后依赖
   - 目标是尽量缩短整体处理时长
   - 允许出现部分失败并在后续补偿

2. **不建议直接并行的场景**
   - 当前元素依赖前一个元素输出
   - 业务必须严格按顺序执行
   - 涉及共享状态更新且无法解耦

3. **处理建议**
   - 先进行流程拆分，将“必须顺序执行”的部分前置或后置
   - 在批量子图内保持无共享状态、无顺序依赖
   - 如果不确定，先用较小并发（如 `2~5`）验证，再逐步上调

## 使用方法

### 输入参数

【**批量执行（并行）**】节点包含三个关键输入：

1. **数组（必填）**
   - 接收数组类型输入（字符串/数字/布尔/对象数组均可）

2. **并发上限（必填）**
   - 控制同一时刻最多并行执行的元素数
   - 建议根据模型/工具速率和资源上限配置

3. **单轮报错重试次数（必填）**
   - 每个元素失败后的重试次数
   - 超过次数后该元素标记为失败并写入 `message`

### 参数调优建议

1. **并发上限**
   - 建议从小值开始（如 `2` 或 `5`），观察模型速率限制与错误率
   - 在无明显限流/超时报错后再逐步提高并发

2. **单轮报错重试次数**
   - 瞬时错误（网络抖动、偶发超时）建议配置 `1~2`
   - 逻辑错误（提示词错误、参数错误）增加重试通常无收益，应优先修复流程本身

3. **数组规模**
   - 大数组建议先拆批处理，避免单次任务过大带来的排队和资源峰值

### 循环体配置

并行节点内部仍通过“开始/结束”定义子流程：

- 【开始】节点提供当前元素值与下标
- 【结束】节点指定当前元素要聚合的输出变量

> 建议将循环体设计为“无共享状态、无顺序依赖”的纯处理流程，以充分发挥并行优势。

## 输出说明

并行批量执行提供三类核心输出：

1. **批量执行结果（成功）**
   - 仅包含成功元素的结果
   - 顺序按输入数组下标排列

2. **批量执行结果（原始）**
   - 每个元素一条记录，格式如下：

```json
[
  {
    "success": true,
    "message": "",
    "data": {}
  },
  {
    "success": false,
    "message": "error message",
    "data": null
  }
]
```

3. **整体运行状态**
   - `success`：全部元素成功
   - `failed`：全部元素失败
   - `partial_success`：部分成功、部分失败

## 约束与注意事项

为避免并发流程中的状态冲突，批量执行（并行）对子图有以下限制：

1. 不允许在子图中放入以下节点类型：
   - 批量执行（串行）
   - 批量执行（并行）
   - 用户选择
   - 表单输入
   - 变量更新

2. 设计建议：
   - 避免依赖跨元素共享变量
   - 避免依赖“前一项输出影响后一项”的顺序逻辑
   - 对必须顺序处理的逻辑，建议拆分到批量节点外单独执行

3. 常见问题排查：
   - 出现限流/超时：先降低并发上限，再检查上游模型或工具配额
   - 失败率偏高：查看原始结果中的 `message`，区分“临时错误”和“确定性错误”
   - 输出不符合预期：优先检查【结束】节点是否选择了正确的输出变量

## 场景示例

### 示例：并行翻译文本数组

假设输入是一个文本数组 `textArray`，希望并行翻译并保留成功结果：

1. 上游节点生成 `textArray`
2. 批量执行（并行）节点配置：
   - 数组：`textArray`
   - 并发上限：如 `5`
   - 单轮报错重试次数：如 `2`
3. 子图中使用 AI 对话节点进行翻译
4. 在【结束】节点输出翻译文本字段

最终可同时获得：

- 成功翻译结果数组（用于下游主流程）
- 原始执行明细（用于排查失败元素）

### 示例：长文本分段翻译（并行）

当长文本超出模型上下文窗口时，可先分段，再对段落数组并行翻译。

#### 实现步骤

1. 文本预处理与分段
   - 上游通过【代码运行】节点将原始长文本切分为 `chunks` 数组
   - 推荐同时输出分段统计信息，便于后续评估并发配置是否合理

2. 配置批量执行（并行）节点

   ![配置批量执行节点](/imgs/fastgpt-loop-node-example-5.png)

   - 数组输入：选择上一步代码运行节点的输出变量 `chunks`。
   - 并发上限：建议从小值开始（如 `3` 或 `5`），再按模型限流逐步提高。
   - 单轮报错重试次数：建议配置 `1~2`，用于吸收瞬时错误。
   - 循环体内添加一个【代码运行】节点，对源文本进行格式化。
   - 添加一个【搜索词库】节点，将专有名词词库作为知识库，在翻译前进行搜索。
   - 添加一个【AI 对话】节点，使用清晰的翻译提示词，约束术语与风格一致性。
   - 添加一个【代码运行】节点，将【AI 对话】节点最后一轮的翻译结果提取出来。
   - 添加一个【指定回复】节点，输出翻译后的文本。
   - 循环体结束节点选择输出变量为【取出翻译文本】的输出变量 `result`。

3. 汇总与校验输出
   - 使用“批量执行结果（成功）”进入下游正文拼接流程
   - 使用“批量执行结果（原始）”排查失败分段并进行补偿
   - 根据“整体运行状态”判断是直接发布还是人工复核
