# FastGPT 排障指南

## 一、常用排障工具

### 1.1 日志查看

#### 应用日志输出
FastGPT 使用分层日志系统，通过环境变量控制日志级别：

**日志级别配置**
```bash
# 控制台日志级别 (debug/info/warn/error)
LOG_LEVEL=info

# 应用日志存储等级（存储到MongoDB）
STORE_LOG_LEVEL=warn

# OTEL（OpenTelemetry）采样日志级别
SIGNOZ_STORE_LEVEL=warn
```

**日志级别说明**
| 级别 | 说明 | 用途 |
|-----|------|------|
| debug | 调试信息 | 开发调试时启用，包含详细过程信息 |
| info | 一般信息 | 记录应用运行状态，默认级别 |
| warn | 警告信息 | 记录潜在问题，如慢查询（>500ms） |
| error | 错误信息 | 记录应用错误和异常 |

**控制台日志输出格式**
```
[日志级别(对齐格式)] YYYY-MM-DD HH:mm:ss 消息 {对象}
例：[Error] 2025-05-05 14:30:45 MongoDB slow query (>2s) {"duration":2500,"collectionName":"chat"...}
```

**常用日志查看命令**

```bash
# Docker 方式查看日志
docker logs <container-name>           # 实时查看
docker logs <container-name> -f        # 持续跟随日志
docker logs <container-name> --tail 100 # 查看最后100行

# Kubernetes 方式查看日志
kubectl logs -f <pod-name>             # 持续跟随日志
kubectl logs <pod-name> --tail=50      # 查看最后50行
```

**MongoDB 日志查询**
- 应用日志存储在 `system_logs` 集合中，字段包括 `text`（消息）、`level`（等级：0=debug/1=info/2=warn/3=error）、`time`（时间）、`metadata`（元数据）
- 只有级别 >= STORE_LOG_LEVEL 的日志才会被存储

#### MongoDB 慢查询日志
MongoDB 操作耗时统计：
- **>2000ms**: 警告级别（通过 `console.warn` 输出到控制台）
- **>500ms**: 警告级别（通过 `console.warn` 输出到控制台）
- 监控的操作：find、save、create、update、delete、aggregate、count、insertMany 等

**注意**：MongoDB 慢查询日志**不会写入 system_logs 集合**，仅通过 `console.warn` 输出到控制台/Docker 日志。如需查看慢查询，请检查容器日志：

```bash
# 查看 Docker 日志中的慢查询
docker logs fastgpt 2>&1 | grep "slow query"

# 查看最后 100 行并筛选慢查询
docker logs fastgpt --tail 100 2>&1 | grep "slow query"
```

### 1.2 数据库连接检查

#### MongoDB 连接检查

**连接参数**
```bash
# 主要环境变量
MONGODB_URI=mongodb://username:password@host:27017/fastgpt?authSource=admin

# 日志数据库（可选，默认同主库）
MONGODB_LOG_URI=mongodb://username:password@host:27017/fastgpt-log?authSource=admin

# 数据库连接池大小
DB_MAX_LINK=100  # 默认100个连接
```

**连接检查命令**
```bash
# 使用 MongoDB CLI 工具连接检查
mongosh "mongodb://username:password@hostname:27017/fastgpt?authSource=admin"

# 执行健康检查命令
> db.adminCommand('ping')
# 返回 { ok: 1 } 表示连接正常

# 查看副本集状态（生产环境推荐）
> rs.status()
```

**Docker 中检查 MongoDB**
```bash
# 进入 MongoDB 容器
docker exec -it fastgpt-mongo mongosh

# 在 MongoDB 中执行
db.adminCommand('ping')
rs.initiate()  # 初始化副本集（如果尚未初始化）
```

**常见 MongoDB 连接错误**
| 错误 | 原因 | 解决方案 |
|------|------|--------|
| `connect ECONNREFUSED` | MongoDB 服务未启动或端口错误 | 检查服务状态，确认 MONGODB_URI 配置 |
| `authentication failed` | 用户名/密码错误或认证库错误 | 验证 MONGODB_URI 中的凭证和 authSource |
| `unable to connect to any servers` | 网络不可达或 DNS 问题 | 检查网络连接，确认 hostname 可解析 |
| `replica set not found` | 副本集未初始化或副本集名称错误 | 检查 docker-compose 初始化脚本或手动执行 rs.initiate() |

#### Redis 连接检查

**连接参数**
```bash
# Redis URL
REDIS_URL=redis://[:password]@hostname:6379

# 集群模式（可选）
REDIS_CLUSTER_NODES=node1:6379,node2:6379,node3:6379
```

**连接检查命令**
```bash
# 使用 redis-cli 检查
redis-cli -h hostname -p 6379 PING
# 返回 PONG 表示连接正常

# 带密码连接
redis-cli -h hostname -p 6379 -a password PING

# 查看 Redis 信息
redis-cli INFO server
```

**Docker 中检查 Redis**
```bash
# 进入 Redis 容器
docker exec fastgpt-redis redis-cli -a mypassword PING

# 查看内存使用
docker exec fastgpt-redis redis-cli -a mypassword INFO memory
```

**Redis 重连机制**
- 自动重试策略：指数退避，最大间隔 2000ms
- 错误重连：READONLY、ECONNREFUSED、ETIMEDOUT、ECONNRESET 时自动重连
- 离线队列：连接断开时缓存命令，重连后执行

#### 向量数据库连接检查

**PGVector (PostgreSQL) 连接**
```bash
# PostgreSQL 连接参数
PG_URL=postgresql://username:password@hostname:5432/fastgpt

# 连接检查
psql "postgresql://username:password@hostname:5432/fastgpt" -c "SELECT version();"
```

**Milvus 连接**
```bash
# Milvus 连接参数
MILVUS_ADDRESS=hostname:19530

# Python 检查示例
from pymilvus import connections
connections.connect("default", host="hostname", port=19530)
connections.get_connection_addr("default")
```

### 1.3 健康检查接口

**系统初始化接口**
```
GET /api/common/system/getInitData
Query: bufferId (可选)
Headers: Authorization: Bearer <token>
```

**接口用途**
- 验证系统状态和配置是否加载
- 获取活跃模型列表和训练任务
- 获取功能开关配置

**预期响应（200）**
```json
{
  "bufferId": "system_buffer_id",
  "feConfigs": { /* 前端配置 */ },
  "systemVersion": "1.0.0",
  "activeModelList": [ /* 可用模型列表 */ ],
  "defaultModels": { /* 默认模型配置 */ },
  "modelProviders": [ /* 模型提供商 */ ],
  "aiproxyChannels": { /* AI代理通道 */ }
}
```

**错误响应说明**
- `401`: 未授权，检查 API Token
- `500`: 服务器错误，查看应用日志获取详情

**OpenAPI 健康检查**
```
GET /api/support/openapi/health
Query: apiKey=<your-api-key>
```

**用途**
- 验证 API Key 是否有效
- 获取关联的 App ID

**预期响应（200）**
```json
{
  "appId": "app_id_123"
}
```

## 二、常见问题与解决方案

### 2.1 应用无法启动

**症状**
- 容器启动失败，立即退出
- 日志中看到初始化错误

**排查步骤**

**第 1 步：检查环境变量**
```bash
# 验证必须的环境变量
echo $MONGODB_URI
echo $REDIS_URL
echo $OPENAI_API_KEY  # 或对应的 AI Provider Key
```

**必要的环境变量清单**
| 变量 | 说明 | 示例 |
|-----|------|------|
| `MONGODB_URI` | MongoDB 连接 | `mongodb://user:pwd@host:27017/fastgpt?authSource=admin` |
| `REDIS_URL` | Redis 连接 | `redis://:password@host:6379` |
| `OPENAI_API_KEY` 或其他 AI Key | AI 模型 API 密钥 | (必填) |
| `PG_URL` | PostgreSQL/向量库 | `postgresql://user:pwd@host:5432/fastgpt` |

**第 2 步：检查数据库连接**
```bash
# 使用上述 1.2 节的命令验证各数据库连接
mongosh "mongodb://..."
redis-cli -h ... PING
psql "postgresql://..."
```

**第 3 步：查看应用启动日志**
```bash
# 本地启动日志
cd FastGPT/projects/app && pnpm dev

# Docker 日志
docker logs -f fastgpt
```

**常见初始化错误**
| 错误 | 原因 | 解决 |
|------|------|-----|
| `MongoError: connect ECONNREFUSED` | MongoDB 未启动 | `docker-compose up -d fastgpt-mongo` |
| `RedisError: ECONNREFUSED` | Redis 未启动 | `docker-compose up -d fastgpt-redis` |
| `database error: no database` | 数据库选择错误 | 检查 MONGODB_URI 中的库名 |
| `connect timeout` | 网络问题 | 检查防火墙、DNS、网络连通性 |

**第 4 步：清理缓存重试**
```bash
# 清理 Node 模块缓存
rm -rf .next build node_modules
pnpm install

# 重启应用
pnpm dev
```

### 2.2 AI 调用失败

**症状**
- 聊天请求返回 5xx 错误
- 日志中看到 LLM 请求失败信息

**排查步骤**

**第 1 步：验证 API Key 配置**
```bash
# 检查环境变量中的 API Key
echo $OPENAI_API_KEY
echo $ANTHROPIC_API_KEY  # 如果使用 Anthropic
```

**检查方式**
- 在系统管理页面验证已配置的模型 Key
- 尝试调用 `/api/common/system/getInitData` 查看 activeModelList

**第 2 步：测试 API Key 有效性**
```bash
# OpenAI 示例
curl -H "Authorization: Bearer $OPENAI_API_KEY" \
  https://api.openai.com/v1/models

# 检查返回状态
# 200 - 有效
# 401 - 无效的 Key
# 429 - 配额不足
```

**第 3 步：检查模型配置**
- 访问应用管理后台 -> AI 模型管理
- 验证模型是否已添加并启用
- 确认模型标识符（model ID）与 API 提供商一致

**常见 AI 调用错误**
| 错误 | HTTP 状态 | 原因 | 解决 |
|------|---------|------|-----|
| `Invalid API Key` | 401 | API Key 无效或过期 | 更新 API Key |
| `Quota exceeded` | 429 | 账户配额用尽 | 检查账户额度或充值 |
| `Model not found` | 404 | 模型标识错误或不可用 | 检查模型 ID 配置 |
| `Timeout` | 504 | API 响应超时 | 增加超时时间或检查网络 |
| `Rate limit exceeded` | 429 | 请求过于频繁 | 实现限流或等待冷却 |

**第 4 步：检查网络和代理**
```bash
# 如果使用代理
echo $HTTP_PROXY
echo $HTTPS_PROXY
echo $NO_PROXY

# 测试代理可用性
curl -x $HTTPS_PROXY https://api.openai.com/v1/models
```

**第 5 步：查看详细错误日志和调用记录**
```bash
# 设置调试日志级别
LOG_LEVEL=debug pnpm dev

# 查看 LLM 请求记录（存储在 llm_request_records 集合）
# 查看应用最近使用时间（存储在 app_records 集合，仅含 lastUsedTime，不含详细调用记录）
```

### 2.3 知识库检索失败

**症状**
- 知识库数据无法检索
- 搜索返回 0 结果，即使存在相关数据
- Embedding 或向量化失败

**排查步骤**

**第 1 步：检查向量存储连接**
```bash
# 验证向量数据库配置
echo $PG_URL  # PGVector
echo $MILVUS_ADDRESS  # 或 Milvus

# 测试连接（以 PGVector 为例）
psql "$PG_URL" -c "SELECT * FROM pg_extension WHERE extname='vector';"
# 应该返回 vector 扩展已安装
```

**第 2 步：验证 Embedding 模型配置**
- 在系统管理 -> AI 模型 -> 向量模型中检查
- 确认默认 embedding 模型已设置
- 验证 embedding API Key 有效

**第 3 步：检查数据训练状态**

`dataset_datas` 集合存储知识库数据记录（字段：q、a、indexes 等），但**不包含** vector（向量数据存储在向量数据库中）、status 字段，也不直接包含 errorMsg。训练状态和错误信息存储在 `dataset_trainings` 集合中。

```javascript
// 查询指定知识库的训练记录（dataset_trainings 集合）
db.dataset_trainings.find({
  "datasetId": ObjectId("你的 dataset_id")
}).sort({ lockTime: -1 }).limit(20)

// 查看训练失败的记录
db.dataset_trainings.find({
  "datasetId": ObjectId("你的 dataset_id"),
  "retryCount": { $lte: 0 },    // retryCount <= 0 表示重试耗尽
  "errorMsg": { $exists: true }  // errorMsg 字段存在于 dataset_trainings 中
}).pretty()

// 查看正在进行中的训练
db.dataset_trainings.find({
  "datasetId": ObjectId("你的 dataset_id"),
  "retryCount": { $gt: 0 }       // retryCount > 0 表示仍在重试中
}).pretty()

// 检查数据是否有对应的 indexes（在 dataset_datas 中）
db.dataset_datas.findOne({ "_id": ObjectId("data_id") })
// 查看 indexes 数组是否为空或缺失 - 如果 indexes: [] 表示未向量化
```

**常见向量化失败原因**
| 原因 | 错误信息 | 解决 |
|-----|--------|------|
| Embedding API 失败 | `Embedding Error: API failed` | 检查 embedding 模型 Key 和网络 |
| 模型不支持 | `Model not found` | 确认模型标识正确 |
| 向量维度不匹配 | `Vector dimension mismatch` | 检查向量库配置的维度 |
| 文本超长 | `Text too long` | 文本过长超过模型限制，自动截断 |

**第 4 步：手动触发重新训练**

通过 API 触发训练重试：
```bash
# 重试集合下所有失败的训练数据
PUT /api/core/dataset/training/updateTrainingData
{
  "datasetId": "...",
  "collectionId": "..."
}

# 重试单条训练数据
PUT /api/core/dataset/training/updateTrainingData
{
  "datasetId": "...",
  "collectionId": "...",
  "dataId": "...",
  "q": "修正后的搜索文本",
  "a": "修正后的答案"
}
```

也可以通过 UI 操作：
1. 选择知识库
2. 进入数据管理
3. 选择异常数据项
4. 点击"重新向量化"或"重新处理"

**第 5 步：检查检索配置**
- 向量搜索参数 `hnswEfSearch`（默认100，越大越精确但慢）
- 搜索结果限制 `assistantRetrievalLimit`
- 相似度阈值配置

```bash
# 验证对象存储环境变量
```

### 2.5 对话流式响应中断

**症状**
- 聊天响应只返回部分内容
- 客户端中断接收流式数据
- 日志中看到生成状态异常

**排查步骤**

**第 1 步：理解生成状态跟踪机制**

FastGPT 使用 `chatGenerateStatus` 追踪聊天消息生成状态（**数值枚举**）：
- `0`（ChatGenerateStatusEnum.generating）: 正在生成中
- `1`（ChatGenerateStatusEnum.done）: 生成完成
- `2`（ChatGenerateStatusEnum.error）: 生成出错

**状态存储位置**
```javascript
// MongoDB chats 集合中（chatGenerateStatus 是 Number 类型）
{
  chatId: "...",
  chatGenerateStatus: 2,   // 0=generating, 1=done, 2=error
  hasBeenRead: false,
  updateTime: ISODate(...)
}
```

**第 2 步：检查流式响应中断原因**
```bash
# 查看错误日志
LOG_LEVEL=debug docker logs fastgpt | grep -i "error\|stream\|generating"

# 检查网络连接
# 确认客户端未主动断开连接

# 检查超时配置
# FastGPT/projects/app/data/config.json 中的超时设置
```

**常见中断原因**
| 原因 | 表现 | 解决 |
|------|------|------|
| LLM API 超时 | 响应不完整 | 增加超时时间配置 |
| 网络不稳定 | 随机中断 | 检查网络质量，实现重试机制 |
| 服务器资源不足 | 长请求经常中断 | 增加服务器资源或实现并发限制 |
| 数据库写入失败 | 中间消息丢失 | 检查 MongoDB 连接和磁盘空间 |

**第 3 步：恢复被中断的对话**
```javascript
// 使用 updateChatGenerateStatus 恢复状态
// 注意 chatGenerateStatus 是 Number 类型，不是字符串
// 0 = generating, 1 = done, 2 = error

db.chats.updateOne(
  { appId: "app_id", chatId: "chat_id" },
  {
    $set: {
      chatGenerateStatus: 1,   // 设置为 done (Number)
      updateTime: new Date()
    }
  }
)
```

**第 4 步：前端处理建议**
- 实现自动重试机制
- 保存中间结果，支持断点续传
- 实现客户端超时退避策略

## 三、性能问题排查

### 3.1 对话响应慢

**排查顺序**

**1. 检查 LLM Provider 延迟**

## 四、数据问题排查

### 4.1 知识库数据异常

**重要说明**：知识库数据分为两层存储：
- `dataset_datas` 集合：存储数据记录（q、a、indexes 等字段），**不包含** vector、status、errorMsg
- `dataset_trainings` 集合：存储训练记录（retryCount、errorMsg 等字段）
- 向量数据存储在向量数据库（PGVector/Milvus）中，不在 MongoDB

**检查训练失败的数据**
```javascript
// 查询训练失败的数据（dataset_trainings 集合）
db.dataset_trainings.find({
  "datasetId": ObjectId("..."),
  "retryCount": { $lte: 0 }

// 查看具体错误信息（errorMsg 在 dataset_trainings 中）
db.dataset_trainings.findOne({
  "_id": ObjectId("training_record_id")

// 查看 errorMsg 字段

// 通过 dataset_datas 的 indexes 字段判断向量化状态
db.dataset_datas.findOne({ "_id": ObjectId("data_id") })
// indexes: [] (空数组) = 未向量化
// indexes: [{...}] (非空) = 已向量化
```

**重新训练流程**

## 四（续）、消息队列问题

### 4.2 训练队列卡住

**症状**
- 知识库数据长时间处于"训练中"状态
- 新建知识库不进行向量化

**排查步骤**

**第 1 步：检查队列状态**
```javascript
// 查看被锁住的训练记录（lockTime 更新时间超过正常值）
db.dataset_trainings.find({
  "datasetId": ObjectId("dataset_id"),
  "retryCount": { $gt: 0 },
  "lockTime": { $lt: new Date(Date.now() - 600000) }  // 锁超过 10 分钟
}).limit(20)

// 查看所有待处理的训练记录
db.dataset_trainings.find({
  "retryCount": { $gt: 0 }
}).sort({ lockTime: 1 }).limit(20)
```

**第 2 步：释放死锁的训练记录**
```javascript
// 将卡住的记录解锁（重置 lockTime）
db.dataset_trainings.updateMany(
  {
    "datasetId": ObjectId("dataset_id"),
    "retryCount": { $gt: 0 },
    "lockTime": { $lt: new Date(Date.now() - 600000) }
  },
  {
    $set: { lockTime: new Date('2000/1/1') }
  }
)
```

**第 3 步：检查 Worker 进程与训练队列**
```bash
# 确认训练 Worker 是否在运行
docker logs fastgpt 2>&1 | grep -i "training\|train\|worker"
```

FastGPT 的训练队列基于 MongoDB change stream 驱动，不使用 Redis/BullMQ。训练数据存储在  集合中。

```javascript
// 检查等待中的训练任务数量（MongoDB 查询）
db.dataset_trainings.countDocuments({ retryCount: { $gt: 0 } })

// 查看训练队列详情
db.dataset_trainings.find({ retryCount: { $gt: 0 } }).sort({ updateTime: -1 }).limit(10)
```

**第 4 步：重启训练队列**
```bash
# 重启 FastGPT 应用以重新初始化训练队列
docker restart fastgpt

# 检查重启后训练是否恢复
docker logs fastgpt --tail 50 2>&1 | grep -i "training"
```

### 4.3 会话队列/SSE 连接问题

**症状**

## 五、部署相关排障

### 5.1 服务启动顺序

在 docker-compose 中，服务依赖关系如下：

**关键服务启动顺序**
```
1. MongoDB (依赖：存储数据)
2. Redis (依赖：缓存/队列)
3. PostgreSQL + pgvector (依赖：向量存储)
4. MinIO (依赖：文件存储)
5. FastGPT 主应用
6. 代码沙箱 (code-sandbox)
7. MCP 服务器
8. 插件服务 (plugin)
```

**使用 docker-compose 启动**
```bash
# 启动全栈开发环境
docker-compose -f FastGPT/deploy/dev/docker-compose.yml up -d

# 验证所有服务健康
docker-compose -f FastGPT/deploy/dev/docker-compose.yml ps

# 查看特定服务日志
docker logs fastgpt-mongo
docker logs fastgpt-redis
```

### 5.2 健康检查配置

**Docker 健康检查示例**
```yaml
healthcheck:
  test: ['CMD', 'pg_isready', '-U', 'username']
  interval: 5s          # 检查间隔
  timeout: 5s           # 超时时间
  retries: 10           # 重试次数
  start_period: 30s     # 启动期限
```

**生产环境建议**
- `interval`: 30s（避免过频检查）
- `retries`: 3-5（容许短暂抖动）
- `start_period`: 根据服务启动时间调整（MongoDB 约 30-60s）

### 5.3 常见部署问题

| 问题 | 原因 | 解决 |
|-----|------|------|
| 容器启动失败 | 资源不足或端口占用 | `docker-compose down && docker-compose up -d` |
| 服务间通信失败 | 网络配置或 DNS 问题 | 检查 docker-compose 中的 networks 配置 |
| 数据卷挂载失败 | 权限或路径问题 | 检查卷权限和路径配置 |
| 环境变量未生效 | docker-compose 缓存 | `docker-compose pull && docker-compose up -d` |

### 6.1 聊天数据排查

```javascript
// 查询特定用户的所有聊天
db.chats.find({

db.chats.find({ 
  "appId": ObjectId("app_id")

db.chats.find({

})

// 查询正在生成的聊天（超过1小时未完成，可能卡住）
db.chats.find({
  "chatGenerateStatus": 0,
  "updateTime": { $lt: new Date(Date.now() - 3600000) }
})
```

**查询聊天消息**
```javascript
// 查询特定聊天的所有消息（chatitems 集合，按 time 字段排序）

// 查询用户提问和 AI 回复（chatitems 集合使用 obj 字段标记角色）

db.chatitems.find({
  "chatId": "chat_id",
```

### 6.3 知识库数据排查

**查询知识库信息**
```javascript

db.datasets.find({ "teamId": ObjectId("team_id") })

db.dataset_trainings.aggregate([
  {
    $match: { "datasetId": ObjectId("dataset_id") }
  },
  {
    $group: {
      _id: "$datasetId",
      totalDocs: { $sum: 1 },
      failedDocs: { $sum: { $cond: [{ $lte: ["$retryCount", 0] }, 1, 0] } }
```
