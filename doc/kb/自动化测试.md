# 自动化测试

> 本文档描述项目的测试框架、规范和运行方式。

## 测试框架

| 类别 | 框架 | 版本 | 说明 |
|------|------|------|------|
| 单元测试 | Vitest | 3.0.9 | 组件和工具函数测试 |
| 覆盖率 | @vitest/coverage-v8 | 3.0.9 | 基于 V8 的代码覆盖率统计 |
| 数据库 Mock | mongodb-memory-server | 10.1.4 | 测试用内存 MongoDB 实例 |

## 运行命令

| 命令 | 用途 | 所属项目 |
|------|------|---------|

## 编写规范

### 文件组织

| 约定 | 说明 |
|------|------|
| 文件命名 | `*.test.ts`（不使用 `*.spec.ts`） |
| 目录位置 | 集中式 `test/` 目录，按模块分层：`test/cases/global/`、`test/cases/service/`、`test/cases/pages/api/` |
| describe 层级 | 最多 2-3 层，按「模块 → 功能 → 场景」组织 |

测试目录结构：

```
test/
├── setup.ts              # 测试启动配置（MongoDB 连接、全局变量初始化）
├── globalSetup.ts        # 全局前置任务
├── mocks/                # 全局 Mock 模块
├── datas/                # 测试数据工具
├── utils/                # 测试辅助工具
├── cases/
│   ├── global/           # packages/global 包测试
│   │   ├── common/       # 通用工具（string, file, math, time 等）
│   │   ├── core/         # 核心逻辑（ai, chat, dataset, workflow 等）
│   │   └── support/      # 支撑功能（permission, user, wallet 等）
│   ├── service/          # packages/service 包测试
│   │   └── common/       # 公共服务（api, cache, file 等）
│   └── pages/api/        # API 路由集成测试
│       └── core/         # 核心 API（evaluation 等）
├── integrationTest/      # 集成测试配置
│   └── vectorDB/         # 向量数据库集成测试
projects/
├── app/test/             # 主应用级测试
└── code-sandbox/test/    # 沙箱服务测试
    ├── unit/             # 单元测试（安全、边界、资源限制等）
    ├── integration/      # 集成测试（API、功能测试）
    └── compat/           # 兼容性测试（旧版 JS/Python）
```

### Mock 策略

项目使用 Vitest 的 `vi.mock` / `vi.mocked` 进行模块级 Mock：

- **Mongoose 模型**：Mock Schema 的 `find`、`findOne` 等静态方法，返回带链式调用（`.sort().lean()`）的 Mock 对象
- **权限鉴权**：Mock `authUserPer` 返回模拟用户鉴权结果，包含 `userId`、`teamId`、`tmbId`、`permission` 等字段
- **数据库连接**：使用 `mongodb-memory-server` 提供内存 MongoDB，在 `beforeAll` 中连接、`afterAll` 中断开，`beforeEach` 中通过 `onTestFinished` 清理数据
- **环境变量**：通过 `vi.stubEnv` 设置测试环境变量（如 `NODE_ENV=test`、`MONGODB_URI`）

### 单测示例

```ts
// 实际文件: test/cases/global/common/string/utils.test.ts
import { describe, expect, it } from 'vitest';
import { getTextValidLength, isObjectId } from '@fastgpt/global/common/string/utils';

describe('string utils', () => {
  it('should calculate valid text length', () => {
    expect(getTextValidLength('a b\nc\t d')).toBe(4);
    expect(getTextValidLength('  \n  ')).toBe(0);
  });

  it('should validate objectId strings', () => {
    expect(isObjectId('507f1f77bcf86cd799439011')).toBe(true);
    expect(isObjectId('507f1f77bcf86cd79943901')).toBe(false);
    expect(isObjectId('507f1f77bcf86cd79943901z')).toBe(false);
  });
});
```

### API 集成测试约定

API 路由测试直接调用 Next.js API handler 函数，Mock 其依赖的 Mongoose 模型和服务层：

- **测试文件位置**：`test/cases/pages/api/`，按业务模块组织子目录
- **鉴权 Mock**：Mock `authUserPer` 返回模拟鉴权结果，覆盖 owner / 非 owner / 鉴权失败等场景
- **断言策略**：验证 handler 返回值结构 + 验证 Mock 函数调用参数（`toHaveBeenCalledWith`）
- **错误场景**：覆盖鉴权失败、数据库查询失败等异常路径

### 沙箱服务测试约定

code-sandbox 测试分为三类：

- **unit/**：安全测试（模块拦截、逃逸攻击、网络安全、文件隔离）、边界测试、资源限制测试
- **integration/**：API 端到端测试、功能集成测试
- **compat/**：旧版 JS/Python 兼容性测试

沙箱测试使用真实的 `ProcessPool` / `PythonProcessPool` 实例，在 `beforeAll` 中初始化进程池。

### 测试配置要点

| 配置项 | 值 | 说明 |
|--------|-----|------|
| fileParallelism | false | 文件级串行执行，避免 MongoDB 冲突 |
| maxConcurrency | 10 | 单文件内最多 10 个测试并发 |
| pool | threads | 使用线程池执行 |
| testTimeout | 20000ms | 单测超时时间 |
| hookTimeout | 30000ms | 钩子超时时间 |
| setupFiles | test/setup.ts | 测试启动文件 |
| globalSetup | test/globalSetup.ts | 全局前置任务 |
