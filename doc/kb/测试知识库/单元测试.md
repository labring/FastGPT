# 单元测试

## 一、测试框架概述

FastGPT 使用 **Vitest** 作为测试框架，这是一个基于 Vite 的高性能单元测试工具，具有与 Jest 兼容的 API。

### 测试框架优势
- **快速执行**：充分利用 Vite 的 ESM 模块加载优化
- **开发友好**：支持热更新（watch mode），开发过程中快速反馈
- **完整兼容**：APIs 与 Jest 完全兼容，支持 describe/it/expect 等常用断言
- **多环境支持**：支持 Node.js、浏览器环境测试

### 测试文件位置

| 位置 | 说明 |
|------|------|
| `FastGPT/test/cases/` | 全局共享代码（packages/）的单元测试 |
| `FastGPT/projects/app/test/` | projects/app 项目特定的单元测试 |
| `FastGPT/projects/code-sandbox/test/` | code-sandbox 项目单元测试 |


### 覆盖率报告

- **生成位置**：`FastGPT/coverage/`
- **报告格式**：HTML、JSON-Summary、JSON
- **生成时机**：每次运行测试时自动生成（除非 cleanOnRerun=false）

## 二、测试运行命令

### 全局命令（根目录执行）

| 命令 | 说明 |
|------|------|
| `pnpm test` | 运行所有测试（包括 test/、projects/*/test/ 中的所有测试） |
| `pnpm test {file-path}` | 运行指定路径的测试文件（支持 glob 模式） |

#### 命令示例

```bash
# 运行所有测试
pnpm test

# 运行特定目录的测试
pnpm test test/cases/global/common/string

# 运行特定测试文件
pnpm test test/cases/global/common/string/string.test.ts

# 运行匹配 pattern 的测试
pnpm test pricing

# Watch 模式（开发时使用，自动重新运行）
pnpm test --watch
```

### 子项目命令

#### projects/code-sandbox/

```bash
cd projects/code-sandbox

# 运行所有测试
pnpm test

# Watch 模式
pnpm test:watch
```

#### projects/app/

projects/app 没有独立的 test 脚本。其测试文件通过根 `pnpm test` 命令执行，测试文件位于 `projects/app/test/` 目录下。

#### packages/diting-rag-ts/

该包有独立的 vitest.config.ts，测试需在其目录内独立执行：

```bash
cd packages/diting-rag-ts
pnpm test
```

## 四、编写测试规范

### 4.1 测试文件命名与位置

#### 命名约定

- **命名后缀**：`.test.ts` 或 `.spec.ts`
- **推荐**：使用 `.test.ts` 作为标准命名
- **文件名**：应与被测试的源文件相同或反映测试功能

#### 示例

```
源文件：packages/global/common/string/tools.ts
测试文件：test/cases/global/common/string/tools.test.ts

源文件：projects/app/src/pages/api/core/evaluation/dataset/collection/create.ts
测试文件：test/cases/pages/api/core/evaluation/dataset/collection/create.test.ts
```

#### 放置位置原则

1. **packages/ 代码的测试**：放在 `test/cases/global/` 下
2. **projects/app/src/pages/api 的测试**：放在 `test/cases/pages/api/` 下
3. **projects/app/src/service 的测试**：放在 `test/cases/service/` 下
4. **项目特定代码**：放在 `projects/{name}/test/` 下

### 4.2 测试结构示例

#### 基础测试结构

```typescript
import { describe, it, expect } from 'vitest';
import { parseFileExtensionFromUrl } from '@fastgpt/global/common/string/tools';

describe('parseFileExtensionFromUrl', () => {
  it('should parse extension from simple URL', () => {
    expect(parseFileExtensionFromUrl('http://example.com/file.pdf')).toBe('pdf');
    expect(parseFileExtensionFromUrl('https://example.com/document.docx')).toBe('docx');
  });

  it('should parse extension from URL with query parameters', () => {
    expect(parseFileExtensionFromUrl('http://example.com/file.pdf?download=true')).toBe('pdf');
    expect(
      parseFileExtensionFromUrl('https://example.com/image.png?size=large&quality=high')
    ).toBe('png');
  });

  it('should return lowercase extension', () => {
    expect(parseFileExtensionFromUrl('http://example.com/file.PDF')).toBe('pdf');
    expect(parseFileExtensionFromUrl('http://example.com/image.JPG')).toBe('jpg');
  });

  it('should handle empty string input', () => {
    expect(parseFileExtensionFromUrl('')).toBe('');
  });
});
```

- 使用 `describe` 组织测试套件
- 每个 `it` 块应测试单一功能
- 清晰的测试描述，说明预期行为
- 使用 `expect` 进行断言

### 4.3 Mock 和 Stub 模式

#### 4.3.1 MongoDB Mock（全局设置）

**位置**：`test/mocks/common/mongo.ts`

**功能**：
- 每个测试创建唯一的隔离数据库（使用 randomUUID()）
- 自动在测试后清理数据库
- 模拟 mongoSessionRun 以避免事务问题

```typescript
import { vi } from 'vitest';
import { randomUUID } from 'crypto';

vi.mock(import('@fastgpt/service/common/mongo/init'), async (importOriginal: any) => {
  const mod = await importOriginal();
  return {
    ...mod,
    connectMongo: async (props: { db: Mongoose; url: string; connectedCb?: () => void }) => {
      const { db, url } = props;
      // 为每个测试创建独立数据库
      await db.connect(url, { dbName: randomUUID(), retryWrites: false });
      await db.connection.db?.dropDatabase();
    }
  };
});

// 模拟 mongoSessionRun，避免测试中使用事务
vi.mock(import('@fastgpt/service/common/mongo/sessionRun'), async () => {
  return {
    mongoSessionRun: vi.fn(async (fn) => {
      return await fn(null as any);
    })
  };
});
```

**在测试中使用：**

### 5.3 全局测试设置（globalSetup.ts）

**位置**：`FastGPT/test/globalSetup.ts`

**功能：**
- 配置 MongoDB 连接（本地开发或 CI/CD）
- 支持 mongodb-memory-server（内存数据库）或外部 MongoDB URI

```typescript
export default async function setup(project: TestProject) {
  const baseUri = process.env.MONGODB_TEST_URI;

  if (baseUri) {
    // 使用外部 MongoDB（CI/CD 或本地开发）
    project.provide('MONGODB_URI', baseUri);
    return async () => {};
  }

  // Fallback：使用内存数据库
  const { MongoMemoryReplSet } = await import('mongodb-memory-server');
  const replset = await MongoMemoryReplSet.create({ replSet: { count: 1 } });
  const uri = replset.getUri();
  project.provide('MONGODB_URI', uri);

  return async () => {
    await replset.stop();
  };
}
```

## 七、最佳实践

### 7.1 编写高质量测试

#### 1. 遵循 AAA 模式（Arrange-Act-Assert）

```typescript
it('should calculate total price correctly', () => {
  // Arrange: 准备测试数据
  const config = {
    priceTiers: [{ maxInputTokens: 30, inputPrice: 1, outputPrice: 2 }]
  };
  
  // Act: 执行被测试的函数
  const { totalPoints } = calculateModelPrice({
    config,
    inputTokens: 20000,
    outputTokens: 10000
  });

  // Assert: 验证结果
  expect(totalPoints).toBe(30); // 20 * 1 + 10 * 2
});
```

#### 2. 每个测试只测试一个功能

```typescript
// 不好 ❌
it('should handle user creation and permissions', () => {
  const user = await createUser('john');
  expect(user._id).toBeDefined();
  
  const perm = await grantPermission(user._id, 'read');
  expect(perm.granted).toBe(true);
});

// 好 ✓
it('should create a user', async () => {
  const user = await createUser('john');
  expect(user._id).toBeDefined();
});

it('should grant permission to user', async () => {
  const user = await createUser('john');
  const perm = await grantPermission(user._id, 'read');
  expect(perm.granted).toBe(true);
});
```

#### 3. 使用有意义的测试描述

```typescript
// 不好 ❌
it('test 1', () => { ... });
it('test pricing', () => { ... });

// 好 ✓
it('should match price tier when input tokens fall within tier range', () => { ... });
it('should apply the highest tier price for tokens exceeding all tier boundaries', () => { ... });
```

#### 4. 避免测试实现细节，测试行为

```typescript
// 不好 ❌ - 测试实现细节
it('should call validatePrice and calculatePrice functions', () => {
  const spy1 = vi.spyOn(priceModule, 'validatePrice');
  const spy2 = vi.spyOn(priceModule, 'calculatePrice');
  getPrice(config);
  expect(spy1).toHaveBeenCalled();
  expect(spy2).toHaveBeenCalled();
});

// 好 ✓ - 测试行为
it('should return correct price for given config', () => {
  const price = getPrice(config);
  expect(price).toBe(expectedPrice);
});
```

## 八、常见问题和排障

### Q1: MongoDB 连接失败

**错误**：`Error: MongoDB connection failed`

**原因**：
- 本地未安装 MongoDB 且 mongodb-memory-server 下载失败
- `MONGODB_TEST_URI` 环境变量配置错误

**解决**：
```bash
# 方案 1：设置外部 MongoDB
export MONGODB_TEST_URI="mongodb://localhost:27017"

# 方案 2：检查网络（mongodb-memory-server 需要下载二进制）
# 将文件提前下载到本地或使用代理

# 方案 3：在 Docker 中运行测试
docker run -it --rm -v $(pwd):/app node:20 bash
cd /app && pnpm install && pnpm test
```

### Q2: 测试超时

**错误**：`Timeout waiting for test to finish`

**原因**：
- 异步操作未正确等待
- 数据库操作过慢
- 外部 API 响应缓慢

**解决**：
```typescript
// 增加超时时间
it('should process large dataset', async () => {
  // 测试代码
}, 60000); // 60 秒超时

// 或在配置中调整全局超时
// vitest.config.mts: testTimeout: 30000
```

### Q3: Mock 不生效

**错误**：实际调用了真实的 MongoDB/Redis 而不是 mock

**原因**：
- Mock 导入路径不匹配
- Mock 在被导入之前定义
- 模块缓存问题

**解决**：
```typescript
// 确保 mock 在测试文件最开始导入
import './mocks'; // 这行必须在最前面
import { describe, it, expect } from 'vitest';

// 或在 test/setup.ts 中统一导入所有 mock
import './mocks/index.ts';
```

### Q4: 数据库污染（一个测试影响另一个）

**错误**：测试顺序改变时结果不同

**原因**：
- 未在 afterEach 中清理数据
- 多个测试共享数据库

**解决**：
```typescript
import { afterEach } from 'vitest';
import { clean } from '@test/datas/users';

afterEach(async () => {
  clean(); // 清理每个测试创建的数据
});
```

### Q5: TypeScript 类型错误

**错误**：`Cannot find module '@test/utils/request'`

**原因**：vitest.config.mts 中的路径别名未正确配置

**解决**：确保 vitest.config.mts 中有：
```typescript
alias: {
  '@test': resolve('test'),
  '@': resolve('projects/app/src'),
  '@fastgpt': resolve('packages')
}
```
