# 本地开发指南

> 本文档描述项目的本地开发环境搭建、启动方式以及 Mock 数据管理策略。

## 环境要求

| 依赖 | 版本要求 | 说明 |
|------|---------|------|
| Node.js | ≥20 | 运行主应用和大多数子项目 |
| pnpm | 9.x（推荐 9.15.9） | 包管理器，monorepo workspace 依赖 |
| Bun | ≥1.x | code-sandbox、mcp_server 子项目的运行时 |
| MongoDB | — | 主数据库 |
| Redis | — | 缓存与消息队列 |
| PostgreSQL / Milvus | — | 向量数据库（二选一或多选） |
| MinIO / S3 兼容存储 | — | 对象存储 |

> **包管理器检测**：项目根目录存在 `pnpm-lock.yaml`，`package.json` 中 `devEngines.packageManager` 指定为 `pnpm@9.15.9`。根目录没有 `.nvmrc` 或 `.node-version` 文件。

## 快速启动

### 1. 安装依赖

```bash
pnpm install
```

项目使用 pnpm workspace monorepo 架构。`pnpm install` 会自动安装所有子包（`packages/*`、`projects/*`、`sdk/*`）的依赖，并执行 `postinstall` 脚本生成 Chakra UI 主题类型定义。

### 2. 配置环境变量

主要开发入口为 `projects/app`。复制模板文件创建本地配置：

```bash
cp projects/app/.env.template projects/app/.env.local
```

**核心配置项说明**：

| 配置项 | 说明 |
|--------|------|
| `MONGODB_URI` | MongoDB 连接串，本地开发建议加 `directConnection=true` |
| `REDIS_URL` 或 `REDIS_CLUSTER_NODES` | Redis 连接（单机或集群模式） |
| `PG_URL` / `MILVUS_ADDRESS` | 向量数据库连接（PG 优先于 OceanBase 优先于 Milvus） |
| `STORAGE_S3_ENDPOINT` | MinIO / S3 对象存储地址 |
| `PLUGIN_BASE_URL` | 插件服务地址 |
| `CODE_SANDBOX_URL` | 代码沙箱服务地址 |
| `AIPROXY_API_ENDPOINT` | AI 代理 API 地址 |
| `FE_DOMAIN` | 前端页面域名（默认 `http://localhost:3000`） |
| `FILE_DOMAIN` | 文件访问域名（通常与 `FE_DOMAIN` 相同） |

完整配置项参考 `.env.template` 文件中的注释说明。

### 3. 启动开发服务器

**主应用**（Next.js 前端 + API）：

```bash
cd projects/app && pnpm dev
```

默认在 `http://localhost:3000` 启动。开发模式下使用 **Rspack** 构建（通过 `next-rspack`），如需切换为 Webpack 构建则设置环境变量 `WEBPACK=1`。

> `pnpm dev` 会先执行 `build:workers` 编译 Worker 线程脚本，然后启动 Next.js 开发服务器。

**代码沙箱服务**（Bun + Hono）：

```bash
cd projects/code-sandbox && pnpm dev
```

**MCP 服务器**：

```bash
cd projects/mcp_server && bun dev
```

### 4. 代理与后端环境配置

本项目是 Next.js 全栈应用，前后端运行在同一进程中，**无需单独配置前端代理**。本地开发时需确保以下后端服务可访问：

| 服务 | 默认地址 | 环境变量 |
|------|---------|---------|
| 插件服务 | `http://localhost:3004` | `PLUGIN_BASE_URL` |
| 代码沙箱 | `http://localhost:3002` | `CODE_SANDBOX_URL` |
| AI 代理 API | `http://localhost:3010` | `AIPROXY_API_ENDPOINT` |
| 对象存储（MinIO） | `http://localhost:9000` | `STORAGE_S3_ENDPOINT` |

以上地址均在 `.env.local` 中配置，可根据实际部署环境调整。

**跨域说明**：`next.config.ts` 中不配置 `ALLOWED_ORIGINS` 时默认允许所有来源跨域请求。

## Mock 策略

### 方案说明

本项目**没有浏览器端的 Mock 系统**（不使用 MSW、vite-plugin-mock、json-server 等）。本地开发时直接连接真实后端服务，通过 `.env.local` 中的服务地址配置指向本地或开发环境的服务实例。

单元测试和集成测试中使用 **Vitest 的 `vi.mock()`** API 对服务端模块进行 Mock，Mock 文件统一存放在 `test/mocks/` 目录下。

### Mock 目录结构（测试用）

```
test/mocks/
├── index.ts                        # Mock 入口，集中导入所有 mock 模块
├── request.ts                      # NextEntry / parseHeaderCert / authCert 等请求层 mock
├── common/                         # 基础设施 mock
│   ├── mongo.ts                    # MongoDB 连接 mock
│   ├── redis.ts                    # Redis 连接 mock
│   ├── bullmq.ts                   # 消息队列 mock
│   ├── s3.ts                       # 对象存储 mock
│   ├── geo.ts                      # 地理位置 mock
│   ├── vector.ts                   # 向量数据库 mock
│   ├── system.ts                   # 系统配置 mock
│   ├── tracks.ts                   # 埋点追踪 mock
│   ├── log.ts                      # 日志 mock
│   └── response.ts                 # 统一响应格式 mock
├── core/ai/                        # AI 核心 mock
│   ├── llm.ts                      # LLM 调用 mock
│   └── embedding.ts                # 向量嵌入 mock
└── support/                        # 业务支撑 mock
    ├── audit/utils.ts              # 审计工具 mock
    └── user/auth/controller.ts     # 用户认证控制器 mock
```

### 如何新增一个测试 Mock

1. 在 `test/mocks/` 对应子目录下创建 `.ts` 文件
2. 使用 `vi.mock()` 定义模块替身，通过 `importOriginal` 保留原始导出的非 Mock 部分：

```typescript
// test/mocks/common/my_service.ts
import { vi } from 'vitest';

vi.mock('@fastgpt/service/xxx/myService', async (importOriginal) => {
  const mod = (await importOriginal()) as any;
  const myFunc = vi.fn(async (params) => {
    return { code: 200, data: { /* mock 数据 */ } };
  });
  return { ...mod, myFunc };
});
```

3. 在 `test/mocks/index.ts` 中添加一行 `import` 注册：

```typescript
import './common/my_service';
```

4. 在测试文件中通过 `beforeAll` 或 setup 文件引入 `test/mocks/index.ts`，Vitest 的 hoisting 机制会自动将 `vi.mock()` 提升到模块加载之前执行

### Mock 数据编写规范

- **响应格式**：API 响应统一使用 `{ code: 200, data: ... }` 结构（由 `request.ts` 中 mock 的 `NextEntry` 包装）
- **权限认证**：通过 `req.auth` 字段注入 mock 用户身份（含 `userId`、`teamId`、`tmbId`、`isRoot` 等），`parseHeaderCert` mock 会从请求中提取该字段
- **数据真实性**：mock 数据应尽量模拟真实数据的字段类型和结构，避免测试假阳性

## 本地调试技巧

### 浏览器 DevTools

- **React DevTools**：项目使用 React 18，建议安装 React Developer Tools 浏览器扩展
- **Source Map**：开发模式下 webpack 使用 `cheap-module-source-map` 策略，可在浏览器 Sources 面板中调试源码

### 构建工具说明

| 模式 | 构建工具 | 说明 |
|------|---------|------|
| 开发模式 | Rspack（默认） | 冷启动和 HMR 更快 |
| 开发模式 | Webpack | 设置 `WEBPACK=1` 环境变量切换 |
| 生产构建 | Webpack | `next build --webpack` |

### 分析构建产物

```bash
# 启用 Bundle Analyzer
cd projects/app && ANALYZE=true pnpm build
```

构建完成后自动打开分析报告页面，可查看各模块体积分布。

### 类型检查

```bash
# 根目录检查所有 TS 文件
pnpm typecheck

# 仅检查 app 项目
cd projects/app && pnpm typecheck
```

### 常用测试命令

```bash
# 运行所有测试
pnpm test

# 运行指定测试文件
pnpm test test/cases/xxx.test.ts

# 运行向量数据库集成测试
pnpm test:vector
```

### 国际化检查

```bash
# 检查三语种词条同步状态
cd projects/app && pnpm check-i18n
```

### 图标管理

```bash
# 初始化图标资源
pnpm initIcon

# 预览所有可用图标
pnpm previewIcon
```
