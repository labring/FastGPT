# 路由模型

> 本文档描述项目的路由架构和路由设计约定，不逐条列出所有路由定义。

## 1. 路由架构

### 整体模式

**Next.js 文件系统路由**：基于 `pages/` 目录结构自动生成路由表，无独立路由配置文件。路由层级由目录嵌套关系决定，页面组件即路由入口。

```
{_app.tsx 全局入口}
└── {NextHead}                ← 全局 Meta 注入
└── {QueryClientContext}      ← React Query Provider
└── {SystemStoreContext}      ← 全局系统配置 Store
└── {ChakraUIContext}         ← UI 主题
└── {Layout}                  ← 主布局（/openapi 除外）
    ├── Auth 守卫层           ← 登录态检查
    ├── Navbar (PC)           ← 左侧 64px 图标导航栏
    ├── NavbarPhone (移动端)  ← 底部 50px Tab 栏
    └── 页面内容
```

### 布局策略

Layout 根据 `router.pathname` 动态选择三种渲染模式：

| 路由类别 | 匹配路径 | 布局行为 |
|----------|---------|---------|
| 无布局路由 | `/openapi` | 跳过 Layout，直接渲染页面 |
| 自定义 Head 路由 | `/chat`, `/chat/share`, `/app/detail`, `/dataset/detail` | 页面自行管理 `<NextHead>`，外层不注入 |
| Dashboard 类页面 | `/dashboard/*`, `/dataset/*`, `/account/*`, `/skill/*`, `/config/*`, `/app/detail` | Auth 包裹，无顶部导航栏（页面自行管理侧边栏） |
| 非标准布局路由 | `/`, `/login/*`, `/chat/share`, `/app/edit`, `/chat`, `/price` | Auth 直接包裹 children，无导航栏 |
| 标准布局（默认） | 其他 | Auth + 左侧 Navbar（PC）/ 底部 NavbarPhone（移动端） |

### 导航结构

PC 端侧边导航栏（`Navbar`）硬编码 4+1 个菜单项：

| 导航项 | 默认链接 | 高亮路径覆盖 |
|--------|---------|-------------|
| Chat | `/chat?appId=...&pane=...` | `/chat` |
| Studio | `/dashboard/agent` | `/dashboard/*`, `/app/detail`, `/skill/detail`, `/dashboard/skill` |
| Datasets | `/dataset/list` | `/dataset/list`, `/dataset/detail` |
| Account | `/account/info` | `/account/*` |
| Plugin | `/config/tool` | `/config/tool`, `/config/tool/marketplace` |

移动端底部导航栏（`NavbarPhone`）的 Studio 高亮路径与 PC 端有差异：不包含 `/dashboard/skill` 和 `/skill/detail`，但额外包含 `/dashboard/tool`、`/dashboard/systemTool`、`/dashboard/templateMarket`、`/dashboard/mcpServer` 及评测相关子路由。

Plugin 菜单项仅对 `username === 'root'` 用户显示。导航栏还支持从后端配置注入外部链接（`feConfigs.navbarItems`），以头像图标形式展示在导航栏底部。

Chat 菜单项通过 `window.open` 在新标签页打开，其他项使用 `router.push` 跳转。

## 2. 懒加载策略

| 策略 | 是否使用 | 说明 |
|------|---------|------|
| `next/dynamic` | 是 | 项目统一使用 `dynamic(() => import(...))` 进行组件级代码拆分 |
| `React.lazy` | 否 | 未使用，全部走 Next.js 的 `dynamic` |
| `{ ssr: false }` | 是 | 依赖浏览器 API 的组件（弹窗、模态框）禁用 SSR |
| `.then()` 具名导出 | 是 | 部分组件使用 `.then((mod) => mod.ComponentName)` 提取具名导出 |

**全局弹窗懒加载**（Layout 中声明，全部 `ssr: false`）：

- `Navbar` / `NavbarPhone`（导航栏，默认 SSR）
- `NotSufficientModal`（余额不足弹窗）
- `SystemMsgModal`（系统消息弹窗）
- `ResetExpiredPswModal`（密码过期弹窗）
- `ImportantInform` / `UpdateContact`（重要通知/更新联系方式）
- `ActivityAdModal`（活动广告弹窗）
- `ManualCopyModal`（手动复制数据，使用 `.then()` 模式）

**页面级组件懒加载**：

各页面组件（如 `dataset/detail`、`dashboard/skill`、`dashboard/tool` 等）将内部的 Tab 面板、表单、列表等子组件通过 `dynamic()` 按需加载，实现页面内的代码拆分。

## 3. 角色权限架构

### 角色模型

FastGPT 使用**自建位域权限模型**（灵感来源于 Linux 文件权限位域），权限值是一个数字，每个 bit 代表一个权限或角色。

**基础角色**（`CommonRoleList`）：

| 角色 | 位值 | 权限映射 |
|------|------|---------|
| read | `0b100` (bit 2) | 读取 |
| write | `0b010` (bit 1) | 写入 + 读取 |
| manage | `0b001` (bit 0) | 管理 + 写入 + 读取 |

**特殊角色**：

| 角色 | 说明 |
|------|------|
| Owner（所有者） | 位值 `~0 >>> 0`（32 位全 1），资源创建者自动获得，拥有所有权限 |
| Null | 位值 `0`，无任何权限 |

**权限模型**: 位域 RBAC（基于角色的访问控制），通过位运算实现权限判断。

### 资源类型与扩展角色

不同资源类型在基础角色之上扩展专属角色：

| 资源 | 扩展角色 |
|------|---------|
| Team（团队） | `appCreate`, `datasetCreate`, `apikeyCreate`, `evaluationCreate`, `skillCreate`, `modelCreate` |
| App（应用） | `readChatLog`（查看聊天记录） |
| Dataset（知识库） | 无扩展，使用基础三角色 |
| Evaluation（评测） | 无扩展，使用基础三角色 |
| Model（模型） | 无扩展，使用基础三角色 |
| AgentSkill | 无扩展，使用基础三角色 |

### 路由-角色绑定

| 绑定方式 | 是否使用 | 说明 |
|----------|---------|------|
| 路由文件按角色拆分 | 否 | 所有路由共用一套页面文件，无 admin/tenant 分文件 |
| 前端路由白名单 | 是 | `auth.tsx` 中 `unAuthPage` 对象定义免登录路由：`/`, `/login`, `/login/provider`, `/login/fastlogin`, `/login/sso`, `/appStore`, `/chat`, `/chat/share`, `/tools/price`, `/price` |
| 菜单项角色过滤 | 是 | `navbar.tsx` 中 `userInfo?.username === 'root'` 条件显示 Plugin 菜单 |
| 动态路由注册 | 否 | 全量路由注册，通过 Auth 组件做守卫拦截 |
| 组件级权限 | 是 | `MemberManager` 组件实现协作者权限管理，按角色展示不同操作入口 |

### 权限判断流程

1. **角色获取**：用户登录后，通过 cookie 中的 token 调用 `/api/support/user/account/tokenLogin` 获取 `UserType`（含 `teamId`、`tmbId`、角色信息等），存入 Zustand `useUserStore`
2. **前端路由守卫**：Auth 组件检查当前路径是否在 `unAuthPage` 白名单中。白名单路由直接放行；非白名单路由调用 `initUserInfo()` 验证 token，验证失败则页面渲染 `null`
3. **API 请求拦截**：所有 API 请求通过 `responseError` 拦截器统一处理。若响应 `code` 属于 `TOKEN_ERROR_CODE`，调用 `clearToken()` 并 `window.location.replace` 跳转到 `/login?lastRoute=...`
4. **后端认证**：`parseHeaderCert()` 函数按优先级尝试三种认证方式——Authorization Bearer（API Key）、Cookie/Token（Session Token）、Root Key（环境变量 `ROOT_KEY`）
5. **业务鉴权**：各资源模块（app/dataset/evaluation 等）的 auth 函数查询 `MongoResourcePermission` 表，获取用户在该资源上的权限记录，通过 `Permission.checkPer()` 位与运算判断是否有目标权限
6. **协作权限继承**：支持文件夹级权限继承——子资源可继承父文件夹的协作者权限，通过 BFS 遍历子树同步
7. **菜单过滤**：前端导航菜单通过运行时条件判断控制可见性（如 root 用户可见 Plugin 菜单）

## 4. 路由守卫逻辑

### 全局守卫流程

FastGPT 的路由守卫逻辑分布在前端 Auth 组件和 API 拦截器两个层面，中间件不做认证（仅开发环境代理）。

**前端 Auth 守卫**（`components/Layout/auth.tsx`）：

1. 读取当前 `router.pathname`
2. 检查是否命中 `unAuthPage` 白名单 → 命中则直接放行
3. 未命中白名单 → 通过 React Query 调用 `initUserInfo()`（即 `getTokenLogin()`）验证 cookie 中的 token
4. `userInfo !== null` → 渲染页面内容
5. `userInfo === null` → 渲染 `null`（空白页），后续由 API 层的 `responseError` 拦截器完成重定向
6. 每 10 分钟自动刷新登录态（`refetchInterval: 10 * 60 * 1000`）

**API 层拦截器**（`web/common/api/request.ts`）：

1. 响应拦截器 `responseError` 检查 `code` 字段
2. Token 错误码 → 清除 token + `window.location.replace` 跳转到 `/login?lastRoute=<当前路径>`
3. 页面例外保护：`/chat/share`、`/price`、`/login` 为外链/登录页，Token 失效和余额不足时均不跳转、不弹窗。`/chat` 仅 Token 失效时受保护（不跳转，保持当前页面），但余额不足时仍会触发弹窗
4. 余额不足错误 → 触发 `NotSufficientModal` 弹窗

**非权限类守卫逻辑**：

| 逻辑 | 实现方式 |
|------|---------|
| 页面标题设置 | `_app.tsx` 中为大多数路由注入统一 `<NextHead>`（含 title/description/favicon）；4 个自定义 Head 路由自行管理 |
| CSP 策略 | `<NextHead>` 中设置 `img-src * data: blob:` |
| Viewport | `<NextHead>` 中固定 `user-scalable=no, viewport-fit=cover` |
| 进度条 | 无（项目未使用 NProgress 等进度条库） |
| UTM 参数处理 | `useInitApp()` 中解析营销参数并存入 sessionStorage，随后清理 URL |

## 5. 路由参数约定

| 约定项 | 说明 |
|--------|------|
| 路径参数 | 前端页面**不使用** Next.js 动态路由（无 `[id].tsx`），所有参数通过 query string 传递 |
| Query 参数 | 广泛使用：Tab/面板切换（如 `?invoiceTab=bill`、`?usageTab=dashboard`）、资源 ID（如 `?appId=xxx`、`?collectionId=xxx`）、分页筛选（如 `?parentId=xxx`）、登录回调（如 `?lastRoute=...`）、分享（如 `?shareId=xxx`） |
| `router.replace` | 用于筛选/参数变更场景，不增加浏览器历史记录（如 Tab 切换 `router.replace({ query: { ...router.query, type: 'new' } })`） |
| `router.push` | 用于页面跳转（如登录成功后跳转 `router.push(lastRoute)`） |
| API 动态路由 | API 层使用 Next.js 动态路由：`[...path].ts` 通配代理路由、`[token].ts` 外链出口路由、`[jwt].ts` JWT 文件访问、`[filename].ts` 文件读取 |

### Query 参数约定

| 参数 | 使用场景 | 示例路由 |
|------|---------|---------|
| `appId` | 应用 ID | `/app/detail?appId=xxx`, `/chat?appId=xxx` |
| `pane` | 聊天面板切换 | `/chat?appId=xxx&pane=HOME` |
| `lastRoute` | 登录后跳转回原页面 | `/login?lastRoute=/dashboard/agent` |
| `shareId` | 分享对话 ID | `/chat/share?shareId=xxx` |
| `collectionId` | 知识库集合 ID | `/dataset/detail?collectionId=xxx` |
| `parentId` | 文件夹导航 | `/dataset/list?parentId=xxx` |
| `invoiceTab` | 账单/发票切换 | `/account/bill?invoiceTab=bill` |
| `usageTab` | 用量面板切换 | `/account/usage?usageTab=dashboard` |
| `teamTab` | 团队面板切换 | `/account/team?teamTab=member` |

### 允许的页面布局自定义

每个页面组件可导出一个 `setLayout` 函数作为自定义布局：

```ts
// 页面组件中导出
export const setLayout = (page: React.ReactElement) => {
  // 自定义布局包装
  return <CustomLayout>{page}</CustomLayout>;
};
```

该函数在 `_app.tsx` 中被调用，替代默认的 children 渲染方式。
