### 2.1 用户注册与认证流程

**入口**

| 操作 | HTTP方法 路径 (file:line) |
|------|-------------------------|
| 密码登录 | `POST /api/support/user/account/loginByPassword` (`projects/app/src/pages/api/support/user/account/loginByPassword.ts:23`) |
| Token登录 | `GET /api/support/user/account/tokenLogin` (`projects/app/src/pages/api/support/user/account/tokenLogin.ts:8`) |
| 登出 | `GET /api/support/user/account/loginout` (`projects/app/src/pages/api/support/user/account/loginout.ts`) |
| 修改密码 | `POST /api/support/user/account/updatePasswordByOld` (`projects/app/src/pages/api/support/user/account/updatePasswordByOld.ts`) |

**流程步骤**

1. 用户通过用户名/邮箱/手机号注册账户，密码在 `UserSchema` 中通过 `hashStr` 自动哈希存储
2. 用户登录后获取 JWT Token，后续所有接口通过 `authCert` 中间件验证身份
3. 权限验证链：`authCert`（基础认证）→ `authUserPer`（用户权限）→ `authApp`/`authDataset`（资源权限）
4. API Key 认证：外部系统通过 API Key 直接调用 `/api/v1/chat/completions`，绕过用户登录态，由 `authOutLinkChatStart` 处理
5. 共享链接认证：通过 `shareId + outLinkUid` 参数，由 `authShareChat` 处理匿名用户访问

**涉及的代码文件**

- `FastGPT/packages/service/support/user/schema.ts` — 用户数据模型（密码哈希、状态管理）
- `FastGPT/packages/global/support/user/team/constant.ts` — 团队成员角色和权限枚举
- `FastGPT/projects/app/src/pages/api/v1/chat/completions.ts` — 认证分支逻辑（共享/团队/API）

**主要异常分支**

| 异常场景 | HTTP状态码 | 触发条件 |
|---------|-----------|---------|
| 密码错误 | 401 | `loginByPassword` 中用户不存在或密码不匹配 |
| Token过期 | 401 | `authCert` 中 JWT Token 验证失败或已过期 |
| 用户被禁用 | 403 | `UserStatusEnum` 为禁用态时拒绝登录 |
| 验证码错误 | 400 | `authCode` 校验登录验证码失败 |

### 2.2 应用创建和发布流程

**入口**

| 操作 | HTTP方法 路径 (file:line) |

| 发布版本 | `POST /api/core/app/version/publish` (`projects/app/src/pages/api/core/app/version/publish.ts`) |

**流程步骤**

1. **创建 App**：调用 `POST /api/core/app/create`，需要团队 `TeamAppCreatePermissionVal` 权限
2. **权限校验**：检查是否有父文件夹写权限（`authApp`）或团队 App 创建权限（`authUserPer`）
3. **类型限制检查**：`checkTeamAppTypeLimit` 验证团队 App/工具数量是否超限
4. **节点安全格式化**：`beforeUpdateAppFormat` 对工作流节点中的 Header 密钥和密码字段加密
5. **数据库事务写入**：在 MongoDB Session 中原子化写入 `MongoApp` + `MongoAppVersion` + `MongoResourcePermission`（Owner 权限）
6. **发布版本**：工作流编辑完成后，调用版本发布接口创建 `MongoAppVersion`（`isPublish: true`），旧版本继续保留以支持回滚
7. **获取最新版本**：`getAppLatestVersion` 优先查最新已发布版本，不存在时降级使用 App 当前 `modules` 字段（草稿态）

**涉及关键 API 和服务**

- `FastGPT/projects/app/src/pages/api/core/app/create.ts` — 创建 App 主逻辑（权限、类型限制、事务写入）

### 2.3 简单应用对话流程

**入口**

| 操作 | HTTP方法 路径 (file:line) |
|------|-------------------------|
| 对话请求(v1) | `POST /api/v1/chat/completions` (`projects/app/src/pages/api/v1/chat/completions.ts:70`) |

**流程步骤**

1. **前端发起请求**：用户在 Web UI 的 ChatBox 组件输入消息，触发 `POST /api/v1/chat/completions`（支持 SSE 流式）
2. **认证分支判断**：
   - 共享链接访问 → `authShareChat`（`shareId + outLinkUid`）
   - 团队空间访问 → `authTeamSpaceChat`（`teamId + teamToken`）
   - API Key / 用户 Token 访问 → `authHeaderRequest`
3. **频率限制**：`teamFrequencyLimit` 检查团队 QPM 限制
4. **获取历史消息**：`getChatItems` 从 MongoDB 查询历史对话，支持多种分页模式（offset / prevId / nextId / initialId）
5. **获取应用版本**：`getAppLatestVersion` 获取应用最新发布的工作流节点配置
6. **工作流调度执行**：`dispatchWorkFlow` 根据节点类型调度执行（AI 对话节点、知识库检索节点等）
7. **LLM 调用**：工作流中的 AI 节点通过 `createLLMResponse` 调用大语言模型，屏蔽流式/非流式差异
8. **流式响应返回**：通过 SSE（Server-Sent Events）将 AI 回复实时推送到客户端，事件类型由 `SseResponseEventEnum` 定义
9. **保存对话记录**：`pushChatRecords` 将用户消息和 AI 回复持久化到 `MongoChatItem`，响应数据存入 `MongoChatItemResponse`
10. **用量记录**：`recordAppUsage` 记录 Token 用量，`updateApiKeyUsage` 更新 API Key 用量

**涉及的代码文件**

- `FastGPT/projects/app/src/pages/api/v1/chat/completions.ts` — 核心对话 API 入口（认证、调度、响应）

- `FastGPT/packages/service/core/ai/llm/request.ts` — LLM 请求核心处理（流式/非流式统一封装、工具调用支持）
- `FastGPT/projects/app/src/components/core/chat/ChatContainer/ChatBox/` — 前端聊天框组件
- `FastGPT/packages/service/core/chat/chatSchema.ts` — Chat 会话数据模型
- `FastGPT/packages/service/core/chat/chatItemSchema.ts` — Chat 消息项数据模型
- `FastGPT/packages/service/core/chat/chatItemResponseSchema.ts` (第47行) — Chat 消息响应数据模型（`MongoChatItemResponse`）

**主要异常分支**

| 异常场景 | HTTP状态码 | 触发条件 |

### 2.4 知识库创建和数据处理流程

**入口**

| 操作 | HTTP方法 路径 (file:line) |
|------|-------------------------|
| 创建知识库 | `POST /api/core/dataset/create` (`projects/app/src/pages/api/core/dataset/create.ts:36`) |
| 创建Collection | `POST /api/core/dataset/collection/create` (`projects/app/src/pages/api/core/dataset/collection/create.ts`) |
| 获取预签名URL | `POST /api/common/file/presignTempFilePostUrl` (`projects/app/src/pages/api/common/file/presignTempFilePostUrl.ts`) |

**流程步骤**

1. **创建知识库**：调用 `POST /api/core/dataset/create`，在 `MongoDataset` 中创建知识库记录，配置 Embedding 模型和分块策略
2. **文件上传**：
   - 前端调用 `POST /api/common/file/presignTempFilePostUrl` 获取 S3 预签名 URL
   - 客户端直接将文件上传到 S3（私有桶），绕过应用服务器，减少带宽消耗
   - 上传前检查团队 `maxUploadFileCount` 限制（30 秒滑动窗口频率限制）
3. **创建 Collection**：调用 `POST /api/core/dataset/collection/create`，在 `MongoDatasetCollection` 中创建集合记录，关联文件 S3 Key
4. **文档解析与分块**：后台 Worker 进程读取 S3 文件，调用解析服务（OCR、PDF 解析等）提取文本，按照分块策略（段落/固定大小）切割为 Chunk
5. **向量化（Embedding）**：每个 Chunk 通过 `getVectorsByText` 调用 Embedding 模型转换为向量，支持批量处理（`batchSize` 控制）和自动分批
6. **存储到向量数据库**：向量和文本数据分别存入向量库（Milvus 或 PGVector）和 MongoDB，通过 `clientManager` 统一管理连接
7. **Training 队列**：分块和向量化任务通过 `MongoDatasetTraining` 异步队列处理，支持失败重试
8. **QA 问答对生成（可选）**：使用 LLM 将文档内容自动转换为问答对格式，提升检索精度

**涉及的代码文件**

- `FastGPT/projects/app/src/pages/api/core/dataset/create.ts` — 创建知识库 API
- `FastGPT/projects/app/src/pages/api/common/file/presignTempFilePostUrl.ts` — 获取 S3 预签名上传 URL
- `FastGPT/packages/service/core/dataset/controller.ts` — 知识库 CRUD（`findDatasetAndAllChildren`、`expandFolderDatasetIds`）
- `FastGPT/packages/service/core/dataset/collection/controller.ts` — Collection 管理（创建、删除关联资源）
- `FastGPT/packages/service/core/ai/embedding/index.ts` — 文本向量化（`getVectorsByText`，支持批量和指令前缀）
- `FastGPT/packages/service/core/dataset/database/clientManager.ts` — 向量数据库客户端管理
- `FastGPT/packages/service/core/dataset/schema.ts` — 知识库数据模型
- `FastGPT/packages/service/core/dataset/data/schema.ts` — 知识数据项模型

**主要异常分支**

| 异常场景 | HTTP状态码 | 触发条件 |
|---------|-----------|---------|
| 知识库数量超限 | 403 | `checkTeamDatasetLimit` 检测团队知识库数量已达上限 |
| 文件大小超限 | 413 | 上传文件超过 `maxUploadFileSize` 限制 |
| S3预签名URL生成失败 | 500 | S3 服务不可用或凭证配置错误 |

### 2.5 知识库检索流程

**入口**

| 操作 | HTTP方法 路径 (file:line) |
|------|-------------------------|
| 检索执行 | `POST /api/core/dataset/search` (`projects/app/src/pages/api/core/dataset/searchTest.ts`) |

**流程步骤**

1. **查询向量化**：将用户提问文本通过 `getVectorsByText` 转换为查询向量
2. **检索执行**（支持三种模式并行）：
   - **向量检索（语义检索）**：在向量数据库（Milvus/PGVector）中按余弦相似度检索
   - **全文检索**：基于 MongoDB Text Index 或 PostgreSQL 全文索引的关键词匹配
   - **混合检索**：同时执行向量和全文检索，通过 RRF（倒数排名融合）算法合并结果
3. **集合标签过滤**：支持按 Collection 标签过滤检索范围，缩小检索集合
4. **重排序（Rerank）**：检索结果通过 `reRankRecall` 调用重排序模型精排，返回 `relevance_score`，按分数截断低相关结果
5. **Agentic Search（多轮检索）**：Agent 模式下 AI 可自主判断检索结果是否充分，决定是否追加检索或调整查询词
6. **结果格式化**：检索结果附加引用来源元数据（Collection 名称、页码等），返回给 AI 节点作为上下文

**涉及的代码文件**

- `FastGPT/packages/service/core/ai/embedding/index.ts` — 查询向量化（`getVectorsByText`）
- `FastGPT/packages/service/core/ai/rerank/index.ts` — 重排序（`reRankRecall`，支持指令前缀格式化）
- `FastGPT/packages/service/core/dataset/database/clientManager.ts` — 向量数据库检索执行
- `FastGPT/packages/service/core/dataset/data/controller.ts` — 知识数据项格式化（`formatDatasetDataValue`，处理图片、S3 URL）
- `FastGPT/packages/service/core/dataset/collection/schema.ts` — Collection 元数据（标签、权限）

**主要异常分支**

| 异常场景 | HTTP状态码 | 触发条件 |
|---------|-----------|---------|
| 向量数据库连接失败 | 500 | Milvus/PGVector 服务不可达或 `clientManager` 连接池耗尽 |
| Embedding模型不可用 | 500 | `getVectorsByText` 中 Embedding 服务返回错误或超时 |
| 检索结果为空 | 200 | 正常返回空列表，由调用方决定是否降级处理 |
| Rerank模型调用失败 | 500 | `reRankRecall` 中重排序服务不可用，检索降级使用原始排序 |

### 2.6 工作流编排和执行流程

**入口**

| 操作 | HTTP方法 路径 (file:line) |
|------|-------------------------|
| 工作流调试 | `POST /api/core/workflow/debug` (`projects/app/src/pages/api/core/workflow/debug.ts`) |
| 代码优化 | `POST /api/core/workflow/optimizeCode` (`projects/app/src/pages/api/core/workflow/optimizeCode.ts`) |

**流程步骤**

**编排阶段（前端）**

1. 用户在可视化画布上拖拽节点（AI 对话、知识库检索、HTTP 请求、代码运行等），通过连线定义执行顺序
2. 工作流以有向图（DAG）结构存储，节点数组（`modules`）+ 边数组（`edges`）持久化到 `MongoApp`

**执行阶段（后端）**

3. 调用 `dispatchWorkFlow` 加载运行时节点（`storeNodes2RuntimeNodes`、`storeEdges2RuntimeEdges`）
4. 根据节点类型（`FlowNodeTypeEnum`）动态路由到对应的 dispatch 处理器（如 `dispatchRunAgent`、`dispatchDatasetSearch` 等）
5. 节点按拓扑顺序执行，每个节点的输出通过边定义传递给下游节点作为输入
6. 支持条件分支（IF-ELSE）、循环（Loop）、并行执行等复杂控制流
7. 工作流最大执行次数受 `WORKFLOW_MAX_RUN_TIMES` 常量限制，防止无限循环
8. 每个节点执行结果记录到 `DispatchNodeResponseKeyEnum.nodeResponse`，用于调试日志展示

**涉及的代码文件**

- `FastGPT/packages/global/core/workflow/node/constant.ts` (第135行) — 工作流节点类型枚举定义（`FlowNodeTypeEnum`，含30+种节点类型）
- `FastGPT/packages/global/core/workflow/runtime/type.ts` — 工作流运行时类型（`RuntimeNodeItemType`、`NodeEdgeGroups`、`ChatDispatchProps` 等）
- `FastGPT/packages/global/core/workflow/runtime/utils.ts` — 运行时工具（`storeNodes2RuntimeNodes`、拓扑排序、变量引用替换）
- `FastGPT/packages/service/core/workflow/dispatch/index.ts` — 工作流主调度器（`dispatchWorkFlow`，节点DAG执行引擎）
- `FastGPT/packages/service/core/workflow/dispatch/ai/agent/index.ts` — Agent 节点执行引擎
- `FastGPT/projects/app/src/pages/api/v1/chat/completions.ts` — 工作流执行入口（通过 dispatchWorkFlow 调度）

**主要异常分支**

| 异常场景 | HTTP状态码 | 触发条件 |
|---------|-----------|---------|
| 工作流执行超限 | 500 | 执行次数超过 `WORKFLOW_MAX_RUN_TIMES`，防止无限循环 |
| 节点执行失败 | 500 | 各节点 dispatch 处理器中抛出异常（如 HTTP 请求失败、代码执行错误） |

| 调试技能 | `POST /api/core/agentSkills/debugChat` (`projects/app/src/pages/api/core/agentSkills/debugChat.ts`) |

**技能创建和配置**

1. 调用 `POST /api/core/agentSkills/create`，`createSkill` 写入 `MongoAgentSkills`（`source: personal`，`currentVersion: 0`）
2. 版本管理：技能通过 `MongoAgentSkillsVersion` 管理历史版本，发布新版本时上传技能包（ZIP）到 S3（`uploadSkillPackage`）
3. 沙箱部署：技能代码运行在隔离沙箱（Agent Sandbox）中，分为 `editDebug`（开发调试）和 `session-runtime`（生产运行）两种模式

**工具调用执行**

4. **Agent 节点初始化**：`dispatchRunAgent` 根据 `AGENT_ENGINE` 环境变量选择执行引擎（默认 Plan+Step 模式，`pi` 值使用 PI-Agent）
5. **能力注册**：将已配置的技能和内置工具注册为工具列表（`SkillToolType[]`），通过 `createSandboxSkillsCapability` 创建沙箱技能能力
6. **主工具调用循环**：`masterCall` 执行多轮工具调用推理循环（最大 `MAX_PLAN_ITERATIONS = 10` 轮）
7. **Plan 模式（可选）**：AI 先通过 `dispatchPlanAgent` 生成任务规划（Plan），再按阶段逐步执行，支持二次任务拆解
8. **工具执行**：AI 决策调用哪些工具后，对应的 dispatch 处理器在沙箱内执行技能代码并返回结果
9. **结果汇总**：所有工具调用结果汇总后，由 AI 生成最终回复

| 沙箱执行超时 | 500 | 技能代码在沙箱中执行超过最大允许时间 |
| 工具调用轮次耗尽 | 500 | `masterCall` 达到 `MAX_PLAN_ITERATIONS=10` 轮仍未完成任务 |

### 2.8 文件上传和处理流程

**入口**

| 操作 | HTTP方法 路径 (file:line) |
|------|-------------------------|
| 获取预签名URL | `POST /api/common/file/presignTempFilePostUrl` (`projects/app/src/pages/api/common/file/presignTempFilePostUrl.ts`) |
| 获取头像预签名URL | `POST /api/common/file/presignAvatarPostUrl` (`projects/app/src/pages/api/common/file/presignAvatarPostUrl.ts`) |

**流程步骤**

1. **获取预签名 URL**：前端调用 `POST /api/common/file/presignTempFilePostUrl`，后端验证权限和用量限制（30s 滑动窗口），生成 S3 预签名 PUT URL（有效期 1 小时）
2. **客户端直传 S3**：浏览器使用预签名 URL 直接将文件上传到 S3 私有桶，文件 Key 格式为 `temp/{teamId}/{filename}`，最大文件大小受 `maxUploadFileSize` 限制
3. **文件解析**：
   - 文档文件（PDF/Word/TXT/Markdown）：提取文本内容
   - 图片文件：使用 VLM（视觉语言模型）理解图片内容并生成描述，存入 `imageDescMap`
   - 音视频：使用 STT（语音转文字）模型转录为文本
4. **对话中使用**：用户在对话中上传的文件，内容提取后作为上下文传入工作流，由 `formatFileInput` 处理文件 URL 列表
5. **知识库导入**：文件上传后创建 Collection，进入知识库数据处理流程（见 2.4）

**涉及的代码文件**

- `FastGPT/projects/app/src/pages/api/common/file/presignTempFilePostUrl.ts` — 预签名 URL 生成（权限验证、频率限制、S3 桶操作）
- `FastGPT/packages/service/core/dataset/data/controller.ts` — 数据格式化（图片描述替换、S3 URL 转换）
- `FastGPT/packages/service/core/workflow/dispatch/ai/agent/index.ts` — `formatFileInput` 处理对话中文件

**主要异常分支**

| 异常场景 | HTTP状态码 | 触发条件 |
|---------|-----------|---------|
| 文件类型不支持 | 400 | 上传文件扩展名不在允许列表中 |
| 预签名URL过期 | 403 | 客户端使用超过1小时的预签名URL上传 |
| S3直传失败 | 500 | 客户端到 S3 的网络中断或桶权限不足 |
| 文件解析失败 | 500 | PDF/Word 解析器无法提取文本（如加密文件、损坏文件） |

### 2.10 App 删除流程

**入口**

| 操作 | HTTP方法 路径 (file:line) |
|------|-------------------------|
| 删除App | `GET /api/core/app/del` (`projects/app/src/pages/api/core/app/del.ts:15`) |

**流程步骤**

1. **权限验证**：`authApp` 校验用户是否拥有目标 App 的 `OwnerPermissionVal` 权限
2. **递归查找子App**：`findAppAndAllChildren` 查找目标文件夹下的所有子 App，全部纳入删除范围
3. **软删除标记**：在 MongoDB Session 事务中将所有 App 的 `deleteTime` 设为当前时间
4. **立即停用**：`deleteAppsImmediate` 立即停止 App 关联的后台任务
5. **异步清理**：`addAppDeleteJob` 将清理任务加入删除队列，异步删除关联的对话记录、Collection、向量数据等

**涉及的代码文件**

- `FastGPT/projects/app/src/pages/api/core/app/del.ts` (第15行) — App 删除 API（权限验证、软删除、队列调度）
- `FastGPT/packages/service/core/app/delete/index.ts` — App 删除队列处理（异步清理关联资源）
- `FastGPT/packages/service/core/app/controller.ts` — `findAppAndAllChildren`、`deleteAppsImmediate`
