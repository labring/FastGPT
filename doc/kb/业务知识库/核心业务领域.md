# FastGPT 核心业务领域

## 一、产品定位与用户价值

### FastGPT 是什么

FastGPT 是一个面向企业和开发者的 **AI Agent 构建平台**。用户无需深厚的 AI 专业知识，即可通过可视化的工作流编排方式，快速构建基于大语言模型（LLM）的智能应用，并将私有知识与 AI 能力结合，打造专属的企业 AI 助手或自动化流程。

### 目标用户群体

| 用户类型 | 典型场景 |
|---------|---------|
| 企业技术团队 | 搭建内部知识库问答机器人、客服自动化系统 |
| 产品/运营人员 | 通过简单模式快速创建 AI 助手，无需编写代码 |
| AI 应用开发者 | 使用工作流模式构建复杂多步骤 AI 流程，调用外部 API |
| 平台管理员 | 统一管理团队模型配置、用量配额和权限 |
| API 接入方 | 通过 OpenAI 兼容接口将 FastGPT 应用集成到自有系统 |

### 核心价值主张

1. **私有知识增强**：将企业文档、数据库接入知识库，让 AI 基于真实私有数据回答，而非依赖模型训练记忆，解决"AI 幻觉"问题。
2. **可视化工作流**：以节点拖拽方式编排 AI 推理流程，将复杂的多步骤任务图形化，降低构建门槛。
3. **多模型统一管理**：支持 OpenAI、Anthropic、Azure、本地私有模型等主流 AI 提供商，通过统一接口调用，方便切换和成本管控。
4. **OpenAI 兼容 API**：对外暴露与 OpenAI Chat Completions 兼容的 API，已有 OpenAI 集成的系统可零成本接入。
5. **团队协作**：支持多人团队共享应用和知识库资源，统一权限管理。

### 与其他 AI 平台的差异

- 相比 ChatGPT / Claude 等对话产品：FastGPT 面向构建者，而非终端对话用户，重点在于让企业定制和部署自己的 AI 应用。
- 相比 LangChain 等开发框架：FastGPT 提供完整的平台（含 UI 界面、知识库管理、用户权限），无需从零搭建基础设施。
- 相比 Dify 等同类平台：FastGPT 的工作流节点设计更贴近程序员思维（节点即函数），同时内置 Agent 沙箱代码执行能力。

## 二、核心业务域

### 2.1 应用（App）域

**用户能创建什么**

在 FastGPT 中，一个"应用"（App）是用户构建和发布 AI 服务的核心单元。每个应用拥有独立的工作流配置、对话设置和权限控制。应用可以组织在文件夹中分层管理。

**应用类型**

| 类型枚举 | 用户视角名称 | 说明 |
|---------|-----------|------|
| `simple` | 简易模式 | 无需了解工作流，通过填写表单即可配置 AI 对话助手，适合快速上线 |
| `chatAgent` | Chat Agent | 对话型 Agent，内置多轮推理和工具调用能力 |
| `workflow`（advanced） | 工作流模式 | 通过节点拖拽自由编排，支持知识库检索、条件分支、循环、HTTP 调用等复杂逻辑 |
| `workflowTool`（plugin） | 工作流工具 | 可被其他应用或 Agent 调用的工具型工作流，相当于可复用的子流程 |
| `mcpToolSet` | MCP 工具集 | 将外部 MCP（Model Context Protocol）服务集成为工具集合 |
| `httpToolSet` | HTTP 工具集 | 将外部 HTTP API 集成为工具集合 |

**简易模式 vs 工作流模式**

- **简易模式**：用户填写系统提示词、选择知识库、配置欢迎语等，由平台自动转换为工作流执行。适合不熟悉 AI 技术的业务人员快速搭建。
- **工作流模式**：用户在可视化画布上连接各类功能节点，完全控制每一步的逻辑。适合需要复杂业务流程（如先检索、再分类、再回答）的场景。

**应用发布与共享机制**

- 应用支持版本控制，发布后生成稳定版本，保证线上服务稳定性，开发中的修改不会影响已发布版本。
- 可通过 API Key 对外开放，接入方通过 OpenAI 兼容接口调用。
- 支持发布到应用市场，供团队或公开用户使用。
- 应用可设置定时触发（Scheduled Trigger），实现无人干预的自动执行。

### 2.2 对话（Chat）域

**用户如何与 AI 应用交互**

用户通过 FastGPT 提供的对话界面或 API 接口向应用发送消息，应用按照配置的工作流处理用户输入并返回 AI 回复。支持文本、图片、文件（PDF、Word、Excel、音视频等）多种输入形式。

**多轮对话管理**

- 每次对话会话（Chat Session）对应一个独立的聊天历史记录。
- 工作流中的 AI 节点可配置携带的历史消息条数，控制上下文长度，平衡效果与 Token 成本。
- 支持对话上下文压缩，在长对话中自动压缩历史信息，避免超出模型上下文窗口限制。

**对话历史与反馈机制**

- 所有对话记录持久化存储，用户可查看历史对话。
- 支持用户对 AI 回复进行点赞/踩等反馈，平台管理员可查看反馈数据，用于质量分析。
- 支持自定义反馈（Custom Feedback），开发者可在工作流中嵌入反馈收集节点。
- 对话支持引用来源展示，AI 回答所参考的知识库片段可在界面中追溯查看。

### 2.3 知识库（Dataset）域

**知识库的作用**

知识库为 AI 提供私有领域知识，是 RAG（检索增强生成）能力的核心。用户将企业文档、产品手册、FAQ 等内容上传到知识库，AI 在回答时会先检索相关内容，再基于检索结果生成答案，从而保证回答的准确性和可追溯性。

**知识库类型**

| 类型 | 说明 |
|------|------|
| 文件夹（folder） | 对知识库进行分层组织，便于分类管理，不直接存储数据 |
| 通用知识库（dataset） | 上传本地文件（PDF/Word/TXT/Markdown/Excel 等），是最常用的方式 |
| 网站知识库（websiteDataset） | 输入网站 URL，系统自动爬取并同步网页内容 |
| 外部文件（externalFile） | 引用外部文件链接，系统自动同步外部文件内容 |
| API 数据集（apiDataset） | 通过 API 接口对接外部数据系统（飞书、语雀等） |
| 飞书知识库（feishu） | 直连飞书文档空间，自动同步内容 |
| 语雀知识库（yuque） | 直连语雀知识库，自动同步内容 |
| 企业数据库（database） | 对接 MySQL/PostgreSQL/MSSQL/Oracle/SQLite/DuckDB 等关系型数据库，支持 NL2SQL 查询 |
| 结构化文档（structureDocument） | 以文件为数据库的结构化存储，保留文档的表格结构用于精确查询 |

**知识库创建流程（用户视角）**

```
上传内容 → AI 解析/分块 → 向量化（Embedding） → 存入向量数据库 → 可被 AI 检索使用
```

1. **上传**：支持本地文件上传、URL 链接导入、API 数据源同步、手动输入问答对等方式。
2. **解析与分块**：系统对文档进行 OCR 解析（支持图片理解）、文本提取，并按照配置的策略（段落切割 paragraph / 固定长度切割 size / 按字符数分割 char / ）拆分为知识片段（Chunk）。
3. **向量化**：每个知识片段由 Embedding 模型转换为向量表示，用于语义检索。
4. **检索**：用户提问时，系统将问题向量化后在知识库中进行相似度匹配，支持：
   - **向量检索（语义检索）**：基于语义相似度，能理解同义词和近义表达
   - **全文检索**：基于关键词匹配，精确匹配特定词汇
   - **混合检索**：结合向量和全文检索，取长补短
   - **数据库检索**：针对关系型数据库，生成 SQL 查询
5. **重排序（Rerank）**：检索结果可通过重排序模型进一步精排，提升相关性。
6. **多轮智能检索（Agentic Search）**：支持 Agent 模式的多轮检索，AI 自主判断是否需要追加检索。

**知识库层级结构**

- **Dataset（知识库）**：顶级组织单元，配置 Embedding 模型、分块策略等全局参数。
- **Collection（集合）**：知识库下的数据源单元。根据来源不同分为多种类型：文件（file）、文件夹（folder）、外部文件（externalFile）、链接（link）、虚拟集合（virtual）、API 文件（apiFile）、图片集（images）、数据库表（table）。每个上传的文件或数据源对应一个 Collection。
- **Data（数据项）**：Collection 下的最小知识单元，即每个知识片段（Chunk）。

### 2.4 工作流（Workflow）域

**工作流是什么**

工作流是 FastGPT 中实现复杂 AI 业务逻辑的核心机制。它将一个完整的 AI 处理过程分解为多个功能节点，用户通过可视化画布将节点连接成有向图，系统按连接关系顺序执行，最终产生输出。

**工作流解决什么问题**

- 一次完整的 AI 服务往往不是"问一句答一句"那么简单，可能需要：先检索知识库→再分类用户意图→再决定用哪种模型→最后整合输出。
- 工作流让这些步骤图形化、可配置，而不需要编写代码。

**工作流节点类型**

| 节点类别 | 代表节点 | 用途 |
|---------|---------|------|
| 系统输入 | 流程开始（Start） | 接收用户输入，是每个工作流的入口节点 |
| AI 对话 | AI 对话节点 | 调用 LLM 生成回复，可配置系统提示词、历史消息、温度等参数 |
| AI 功能 | 问题分类（CQ） | 对用户问题进行意图分类，根据分类结果走不同分支 |
| AI 功能 | 内容提取 | 从文本中结构化提取指定字段 |
| 知识库 | 知识库检索 | 在指定知识库中检索相关内容，输出引用片段 |
| 知识库 | 知识库搜索合并 | 合并多个知识库的检索结果 |
| 工具调用 | HTTP 请求 | 调用外部 HTTP/REST API，实现与第三方系统集成 |
| 工具调用 | 代码运行（Sandbox） | 在安全沙箱中执行 JavaScript/Python 代码 |
| 工具调用 | 工具节点 | 调用已配置的 Agent 技能或外部工具 |
| 交互 | 指定回复 | 直接输出固定文本，不经过 LLM |
| 交互 | 用户选择 | 向用户展示选项，等待用户点击后继续流程 |
| 交互 | 表单输入 | 向用户展示表单，收集结构化输入 |
| 流程控制 | 条件判断（IF-ELSE） | 根据变量值进行条件分支 |
| 流程控制 | 循环（Loop） | 对数组数据进行批量循环处理 |
| 流程控制 | 并行运行 | 同时执行多个分支，等待全部完成后汇总 |
| 数据处理 | 文本加工 | 拼接、格式化文本 |
| 数据处理 | 变量更新 | 更新全局变量的值 |
| 数据处理 | 文档读取 | 读取用户上传的文件内容 |
| 系统配置 | 用户引导 | 配置对话框欢迎语、问题引导等 |

**工作流调试与测试**

- 工作流编辑器内置"在线调试"功能，可直接在画布上测试工作流，查看每个节点的输入输出数据。
- 支持单节点调试，精准定位问题节点。
- 提供完整的执行日志（用量统计、耗时、错误信息）。

### 2.5 Agent 技能（AgentSkill）域

**Agent 技能的概念**

Agent 技能是一种独立封装的可复用功能单元，可以被 Agent 节点直接调用。技能将复杂的能力（如搜索引擎查询、代码执行、数据处理等）打包为标准化接口，Agent 可以根据用户意图自主决定调用哪些技能及如何使用。

**技能的来源**

| 来源 | 说明 |
|------|------|
| 系统技能（system） | 平台内置提供的公共技能，如 Bing 搜索、Google 搜索等 |
| 自定义技能（personal） | 团队/用户自行开发和上传的技能，通过打包为技能包（ZIP）导入 |

**技能分类**

| 分类 | 说明 |
|------|------|
| 搜索（search） | 网络搜索、文档检索等 |
| 工具（tool） | 通用工具调用 |
| 编程（coding） | 代码生成、执行 |
| 数据处理（data） | 数据格式转换、处理 |
| 分析（analysis） | 数据分析、图表生成 |
| 通信（communication） | 消息发送、邮件等 |
| 其他（other） | 不在上述分类的技能 |

**技能沙箱机制**

技能的代码在隔离的沙箱环境（Agent Sandbox）中执行，保障宿主系统安全。沙箱支持两种模式：
- **编辑调试沙箱**（edit-debug）：技能开发者调试技能时使用。
- **会话运行时沙箱**（session-runtime）：生产环境中 Agent 实际调用技能时使用。

**技能版本控制**

每个技能支持版本管理，发布新版本时旧版本可回滚，保证生产环境的稳定性。

### 2.6 模型管理域

**支持的 AI 模型提供商**

FastGPT 通过统一的 AI 接口层支持主流 AI 服务提供商，包括但不限于：

| 提供商类型 | 典型代表 |
|---------|---------|
| 通用大模型 | OpenAI（GPT-4o、o3 等）、Anthropic（Claude）、Google Gemini |
| 国内大模型 | 阿里通义千问、百度文心、智谱 AI、月之暗面、DeepSeek 等 |
| 私有化/本地部署 | Ollama、vLLM 等本地服务（兼容 OpenAI 接口） |
| Azure | Azure OpenAI Service |

**模型用途分类**

FastGPT 中的模型分用途配置：

| 模型类型 | 用途 |
|---------|------|
| LLM（对话模型） | AI 对话节点的生成，工具调用，内容理解；支持 `vision` 属性标记是否具备图片理解能力（视觉语言能力） |
| Embedding（向量模型） | 将文本转为向量，用于知识库语义检索 |
| Rerank（重排序模型） | 对检索结果精排，提升检索准确率 |
| STT（语音转文字） | 语音输入转录，Whisper 等 |
| TTS（文字转语音） | AI 回复语音播报 |

**如何配置和切换模型**

- 平台管理员在系统后台统一配置可用模型列表（模型 ID、API Key、价格、能力限制等）。
- 应用/工作流创建者在节点中从已配置的模型列表中选择使用的具体模型。
- 系统支持按用量计费，管理员可为不同模型配置 Token 价格（支持阶梯定价），追踪团队用量。

## 三、用户角色与权限

| 角色 | 权限范围 | 说明 |
|------|---------|------|
| 平台管理员 | 全局系统配置、模型管理、用量监控、团队管理 | 部署 FastGPT 的运维人员，负责系统级配置 |
| 团队 Owner | 团队内所有资源的创建、编辑、删除、共享 | 创建团队的用户，拥有团队最高权限 |
| 团队成员 | 根据具体资源权限配置，可读/写/管理团队内资源 | 被邀请加入团队的用户，权限由 Owner 控制 |
| API 用户 | 通过 API Key 调用特定应用的对话接口 | 外部系统/开发者，无平台 UI 访问，仅限 API 调用 |

**权限说明**

- 知识库、应用、Agent 技能均支持独立的权限配置，可精细控制团队成员的可见和可操作范围。
- 资源支持"继承权限"（子资源自动继承父文件夹权限）和"独立权限"两种模式。
- 团队成员状态：活跃（active）、已离队（leave）、已禁用（forbidden）。

## 五、关键业务术语解释

| 术语 | 解释 |
|------|------|
| App | FastGPT 中用户创建的 AI 应用，是对外提供服务的基本单元，包含工作流配置和对话设置 |
| Dataset | 知识库，用于存储私有文档/数据并提供语义检索能力，给 AI 补充私有领域知识 |
| Collection | 知识库中的集合，对应一个数据源（一个文件、一个网页、一个 API 接口等） |
| Chunk（Data） | 知识库中的最小知识单元，即被分割后的文本片段，是实际参与向量检索的基本单位 |
| AgentSkill | Agent 技能，封装了特定功能（如搜索、代码执行）的可复用能力包，可被 Agent 自主调用 |
| Workflow | 工作流，由节点和连线组成的有向图，描述 AI 处理用户请求的完整流程 |
| Flow Node | 工作流节点，工作流中的基本功能单元，每个节点代表一个处理步骤（如 AI 对话、知识库检索、HTTP 调用等） |
| API Key | API 访问密钥，外部系统通过 API Key 调用 FastGPT 应用，无需登录平台界面 |
| Embedding | 将文本转换为高维向量的过程，是知识库语义检索的基础技术 |
| RAG | 检索增强生成（Retrieval-Augmented Generation），先从知识库检索相关内容，再将其作为上下文传给 AI 生成回答，提升准确性和可溯源性 |
| Rerank | 重排序，对初步检索结果使用专用模型再次精排，提升最终召回内容的相关性 |
| Plugin（工作流工具） | 可被其他应用调用的工作流，相当于可复用的 AI 子程序 |
| MCP | Model Context Protocol，Anthropic 提出的模型上下文协议，FastGPT 支持接入 MCP 服务器提供的工具 |
| Team | 团队，用于组织多个成员共享 App、Dataset 等资源，统一管理权限 |
| TmbId | 团队成员 ID（Team Member Id），用于标识操作者身份，进行权限验证和操作审计 |
| Sandbox | 沙箱，隔离的代码执行环境，Agent 技能和代码运行节点在此安全执行代码，防止对主系统造成影响 |
