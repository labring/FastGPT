# FastGPT 企业内部使用功能缺口规划

## 目标

基于当前 fork `enterprise/internal-hardening`，盘点 FastGPT 社区版用于企业内部正式落地时缺少的能力，并规划后续强化顺序。

本文件关注企业内部真实上线，不只关注能否跑起来。

## 当前已有能力

社区版已经适合做 POC 和部门级验证：

1. 应用工作流编排。
2. 分享链接和 OpenAPI。
3. 知识库/RAG。
4. 多模型配置。
5. 模型日志看板。
6. 对话、知识库、应用等核心数据结构。
7. Docker Compose 自托管部署。
8. 本 fork 已新增企业部署安全 env 样板和部署前检查脚本。

## 当前主要缺口

### 1. 企业级身份与成员体系

缺口：

1. 社区版缺少 SSO 登录。
2. 社区版缺少外部成员同步。
3. 社区版缺少团队空间与权限管理的完整企业功能。
4. 社区版缺少管理后台。

影响：

1. 无法直接接入公司 IdP、飞书、企业微信、钉钉、SAML、OAuth。
2. 离职、转岗、组织架构变化无法自动同步。
3. 无法做可靠的部门级访问隔离。
4. root 账号和本地账号管理会变成运维风险。

规划：

1. 优先评估商业版 SSO/团队权限是否满足需求。
2. 若坚持社区版二开，先做最小企业身份桥接：OAuth/OIDC 登录、禁用自注册、单团队模式、用户白名单。
3. 中期补组织架构同步、成员停用、权限回收。

优先级：P0

### 2. 企业部署 runtime 配置

缺口：

1. 官方 compose 示例仍包含大量弱默认值。
2. 当前 `deploy/enterprise/.env.example` 只是安全基线，还没有直接驱动 runtime compose。
3. 暴露端口、Docker socket、MinIO console、MCP server、沙盒服务需要按企业环境收敛。

影响：

1. 部署人员容易照抄示例密码。
2. 正式服务可能暴露数据库、对象存储或沙盒控制面。
3. 安全检查脚本和实际部署文件还没有完全闭环。

规划：

1. 新增 `deploy/runtime/docker-compose.enterprise.yml`。
2. 所有密码、token、URL、开关都改为 `${ENV}` 注入。
3. 默认只暴露 FastGPT Web 入口。
4. 将 MinIO、MongoDB、PostgreSQL、Redis、插件、沙盒、MCP、AIProxy 放在内部网络。
5. CI 中强制运行 `pnpm security:check-env`。

优先级：P0

### 3. 审计日志与合规留存

缺口：

1. 社区版有应用运营日志和模型日志，但缺少企业审计视角。
2. 缺少统一记录谁在什么时间访问、修改、删除了应用/知识库/API Key/模型配置。
3. 缺少敏感字段脱敏和日志留存策略。

影响：

1. 出现资料外泄或误操作时难以追溯。
2. 无法满足企业内控、资安、法务审计要求。
3. 管理员操作缺少不可抵赖证据。

规划：

1. 定义审计事件模型：登录、登出、创建、修改、删除、权限变更、API Key 创建/删除、知识库导入/删除、应用发布、模型配置变更。
2. 增加审计事件写入服务。
3. 输出到 MongoDB 专用集合和 OTEL/log pipeline。
4. 增加敏感字段脱敏。
5. 建立保留周期和导出格式。

优先级：P1

### 4. 权限与数据隔离策略

缺口：

1. 社区版不提供完整团队空间与权限能力。
2. 应用、知识库、API Key、对话日志之间的访问边界需要企业化确认。
3. 缺少按部门、群组、资源类型的默认权限策略。

影响：

1. 知识库可能被错误团队或错误应用调用。
2. 对话日志可能包含敏感资料却缺少访问限制。
3. 内部 AI 平台难以从小范围 POC 扩到全公司。

规划：

1. 若采购商业版，优先使用商业版团队/成员组/权限。
2. 若二开社区版，先做单团队 + 管理员分级，不立即追求完整多租户。
3. 再补资源级权限：应用、知识库、API Key、对话日志。
4. 增加默认权限模板：全员只读、部门编辑、管理员管理。

优先级：P1

### 5. 知识库同步与治理

缺口：

1. 社区版缺少第三方知识库定时同步。
2. 社区版缺少 web 站点同步。
3. 社区版缺少知识库索引增强、图片知识库等高级能力。
4. 缺少企业文档生命周期治理。

影响：

1. 文档更新后知识库容易过期。
2. 无法稳定接入 Confluence、SharePoint、Google Drive、飞书文档、企业内部 CMS。
3. RAG 质量依赖人工维护，规模化困难。

规划：

1. POC 先手动导入关键文档，建立质量评估集。
2. 第一批同步源选 1 个最重要系统，例如飞书文档或 SharePoint。
3. 设计同步任务：增量扫描、变更检测、失败重试、删除同步、权限映射。
4. 增加知识库健康看板：文档数量、更新时间、失败任务、召回质量样本。

优先级：P1

### 6. 内容安全与模型治理

缺口：

1. 社区版缺少模型内容审核。
2. 缺少企业级 prompt/回答安全策略。
3. 缺少按模型、应用、用户、部门的用量预算和限额。

影响：

1. 可能输出敏感、不当或违规内容。
2. 模型 token 成本难以控制。
3. 无法对高风险应用做更严格策略。

规划：

1. 接入统一模型网关或 AIProxy 策略层。
2. 增加输入/输出内容审核钩子。
3. 增加应用级模型白名单。
4. 增加用户、应用、部门级调用限额。
5. 增加高风险关键词与敏感资料检测。

优先级：P2

### 7. 应用发布安全

缺口：

1. 社区版缺少应用发布安全配置。
2. 社区版缺少第三方应用发布，例如飞书、公众号等。
3. 分享链接、iframe、API 发布需要统一安全策略。

影响：

1. 内部应用可能被无意公开。
2. API Key 或分享链接生命周期不可控。
3. 第三方渠道发布缺少审批与边界。

规划：

1. 增加分享链接有效期、访问密码、访问 IP/CIDR 限制。
2. 增加 API Key 到期时间、用途说明、最后使用时间。
3. 增加发布审批流程：草稿、待审批、已发布、下线。
4. 增加发布清单看板。

优先级：P2

### 8. 生产可观测性与运维

缺口：

1. 当前有 OTEL 相关配置，但缺少完整可观测性 runbook。
2. 缺少备份恢复操作手册。
3. 缺少健康检查、容量规划、升级演练标准。

影响：

1. 服务异常时定位慢。
2. 数据库或对象存储故障时恢复风险高。
3. 升级 FastGPT 或依赖服务时不可预期。

规划：

1. 建立指标：请求量、错误率、延迟、模型调用、队列、知识库导入任务、数据库连接、Redis 内存。
2. 建立告警：服务不可用、数据库不可用、对象存储不可用、沙盒错误率、模型调用错误率。
3. 编写备份恢复 runbook。
4. 建立 staging 升级流程。

优先级：P1

### 9. 企业管理后台

缺口：

1. 社区版缺少管理后台。
2. 缺少全局用户、团队、应用、知识库、模型、API Key、日志的统一管理视图。

影响：

1. 运维人员需要直接查数据库或分散在多个页面操作。
2. 大规模内部推广后管理成本会快速上升。

规划：

1. 优先评估商业版管理后台。
2. 若二开，先做只读运营后台：用户数、团队数、应用数、知识库数、调用量、错误率。
3. 再补管理操作：停用用户、回收 API Key、下线应用、冻结知识库。

优先级：P2

## 推荐路线

### 阶段 0：部署闭环

目标：先让部署安全可控。

交付：

1. `docker-compose.enterprise.yml`
2. `.env.enterprise` 驱动部署
3. `security:check-env` 接入 CI
4. 反向代理和端口收敛说明
5. 备份恢复初稿

### 阶段 1：企业身份与权限决策

目标：决定买商业版还是二开社区版。

交付：

1. SSO/成员同步评估表。
2. 商业版试用或 PoC。
3. 社区版二开成本估算。
4. 单团队/多团队/同步模式选择。

建议：

如果目标是全公司使用，优先采购或试用商业版；自己重做 SSO、团队权限和管理后台成本高，且后续升级维护压力大。

### 阶段 2：审计、知识同步、运维

目标：让平台可追溯、可维护、可扩展。

交付：

1. 审计日志事件模型。
2. 第一个知识库同步源。
3. OTEL/日志/指标接入。
4. 备份恢复演练。
5. 升级演练流程。

### 阶段 3：治理与发布安全

目标：从内部工具升级为企业 AI 平台。

交付：

1. 内容审核。
2. 模型与成本治理。
3. 应用发布审批。
4. API Key 生命周期管理。
5. 管理后台或运营看板。

## 当前最建议先做的 5 件事

1. 做企业版 runtime compose，让 `.env.enterprise` 真正驱动部署。
2. 决定 SSO/权限走商业版还是社区版二开。
3. 建立审计日志最小事件模型。
4. 做备份恢复 runbook。
5. 选一个真实部门做 RAG POC，建立问题集和评估标准。

## 暂不建议先做的事

1. 先大规模改 UI。
2. 先重做完整多租户。
3. 先同时接入很多知识源。
4. 先做复杂计费系统。
5. 在没有明确边界前启用所有沙盒和 MCP 能力。
