# 最小账号治理方案

## 目标

在正式 SSO 或商业版上线前，降低社区版账号体系带来的企业安全风险。

## 原则

1. root 只用于初始化和应急。
2. 普通用户不允许绕过企业流程自行注册。
3. 管理员账号必须有申请、审批、发放、回收记录。
4. 登录失败、管理员操作、API Key 操作必须进入审计。
5. 试点阶段优先单团队，避免误以为社区版已具备完整企业权限隔离。

## 必做配置

| 配置 | 建议 |
| --- | --- |
| `DEFAULT_ROOT_PSW` | 高强度临时密码，初始化后轮换 |
| `PASSWORD_LOGIN_LOCK_SECONDS` | 300 秒或以上 |
| `PASSWORD_EXPIRED_MONTH` | 3 个月 |
| `MAX_LOGIN_SESSION` | 3 或更低 |
| `USE_IP_LIMIT` | `true` |
| `TRUSTED_PROXY_ENABLE` | `true` |
| `TRUSTED_PROXY_IPS` | 只配置反向代理 IP/CIDR |
| `ALLOWED_ORIGINS` | 明确域名，不使用通配 |

## 账号流程

### 账号创建

1. 由业务负责人提交申请。
2. 管理员确认成员、部门、用途。
3. 创建账号或加入团队。
4. 记录申请单编号、创建人、有效期。

### 权限变更

1. 必须说明资源范围：应用、知识库、API Key、模型配置。
2. 必须说明角色：使用、编辑、管理。
3. 变更后写入审计日志。

### 离职或转岗

1. 禁用账号。
2. 回收 API Key。
3. 转移应用和知识库 owner。
4. 检查分享链接和外部发布。
5. 保留审计记录。

## root 账号规范

1. root 密码不共享。
2. root 操作必须有工单或事件编号。
3. root 登录后尽快退出。
4. root 密码定期轮换。
5. 不使用 root 创建日常应用和知识库。

## 社区版短期限制

社区版在企业权限上不应承诺：

1. 完整部门权限隔离。
2. 外部成员自动同步。
3. 全局管理后台。
4. 完整资源审批流程。

试点文档和宣导中需要明确这些边界。

## 验收

1. 部署配置通过 `pnpm security:check-env`。
2. root 密码不再使用默认值。
3. 登录失败和登录成功有审计记录。
4. 管理员账号清单可追溯。
5. 试点成员清单和资料范围明确。
