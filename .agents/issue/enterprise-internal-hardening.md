# 企业内部部署强化问题分析

## 背景

FastGPT 默认部署文件面向快速体验，包含多个演示口令、开放端口和偏开发环境的安全默认值。企业内部正式使用时，通常会接入公司知识库、模型服务、对象存储、SSO 或内部 API，部署基线需要先收紧。

## 当前风险

1. Docker 示例中存在固定默认密钥和弱口令，例如 root 密码、TOKEN_KEY、FILE_TOKEN_KEY、AES256_SECRET_KEY、ROOT_KEY、数据库密码、Redis 密码、MinIO 账号密码、插件 token、沙盒 token、AIProxy token。
2. `USE_IP_LIMIT`、`CHECK_INTERNAL_IP`、`TRUSTED_PROXY_ENABLE`、`ALLOWED_ORIGINS` 等安全开关需要企业部署时显式配置，否则容易出现滥用、SSRF、跨域和客户端 IP 识别风险。
3. 沙盒能力会执行用户或工作流代码，需要明确限制内网访问、模块白名单、请求次数、请求体/响应体大小和执行资源。
4. 示例部署会暴露 FastGPT、对象存储、MCP 等端口。企业内网部署应通过反向代理统一入口，并最小化外露端口。
5. 上线前缺少一个可以本地或 CI 执行的部署配置检查入口。

## 第一阶段目标

先补充企业部署安全基线，不改核心业务流程：

1. 新增企业部署环境变量样板。
2. 新增部署前环境检查脚本。
3. 新增企业内部部署说明。
4. 在根 `package.json` 提供可运行脚本，便于 CI 接入。

## 非目标

1. 不在第一阶段改造 SSO、RBAC、审计日志或商业版功能。
2. 不直接修改官方生成的 Docker Compose 模板，避免影响上游同步。
3. 不替用户生成真实生产密钥；密钥必须由部署方在受控环境中生成和保管。
